## typescript-cop — round 1

Scope: the Phase 6 diff against `main` — `src/client/resources/base-resource.ts`,
`src/schema-overrides/{pagination,device-overrides,alert-overrides,types,write-bodies,index}.ts`,
and their test files (`tests/unit/client/{base-resource,paginate}.test.ts`,
`tests/unit/schema-overrides/*.test.ts`). Also read the uncommitted working-tree diff to
`write-bodies.ts` (the `implementation-auditor-r2-f1` fix, already re-verified by
`implementation-auditor-r3.md`) as the current state of that file. Cross-checked `parseLenient`'s
`Lenient<T>` contract (`src/validation/schema-leniency.ts`) — established in Phase 4 specifically to
give `parseLenient`'s success type an honest shape after `typescript-cop-r1-f1` in
`review-phase4/typescript-cop-r1.md` flagged the same defect this round's finding below reintroduces
at the `BaseResource` layer — and confirmed `z.prettifyError`'s output carries no wire value (only
`issue.message`/path), so the R20 masking invariant on `validateArrayResponse`'s aggregated `warn`
holds.

The dominant concern: `BaseResource.validateResponse`/`validateArrayResponse` cast `parseLenient`'s
honestly-typed `Lenient<T>` result back down to the bare `T` every resource method will declare,
undoing the exact type-safety fix Phase 4 built for this. Everything else checked out clean.

## Findings

| ID | Severity | Status | Category | Location | Finding | Recommendation |
|----|----------|--------|----------|----------|---------|----------------|
| typescript-cop-r1-f1 | High | Open | TypeHole | `src/client/resources/base-resource.ts:265-275` (`validateResponse`), `:298-343` (`validateArrayResponse`) | Both methods call `parseLenient(schema, data, this.logger, context)` — and because `this.logger` is a required constructor param, this always selects the overload that returns `z.ZodSafeParseResult<Lenient<T>>` (`schema-leniency.ts:758-763`). `Lenient<T>` exists specifically because Phase 4's own `typescript-cop-r1-f1` (`review-phase4/typescript-cop-r1.md`) found that the bare, unmapped `T` "quietly lies" about nullability: every named field, at every depth, independently tolerates `null` on the wire regardless of what `T` (the generated/override type) declares. `validateResponse` does `return result.data as T;` and `validateArrayResponse` does `items.push(result.data as T);` — both re-collapse the honest `Lenient<T>` back onto the narrower `T` that every Phase 7/8 resource method will declare as its return type (`Promise<Device>`, `Promise<Alert>`, …). The consequence is exactly the regression Phase 4 fixed: a real `{ hostname: null }` response validates successfully and is handed to a caller as a `Device` whose `hostname` is typed `string` (or `string | undefined` — never `null`), so the idiomatic `if (device.hostname) { device.hostname.toUpperCase() }` a consumer writes against the declared type throws `TypeError: Cannot read properties of null` at runtime on data the type claimed was excluded. Unlike the enum-widening leniency (which Phase 2's codemod honestly reflects in the generated type, and which `types.ts`'s `Omit`/`Pick` graft explicitly re-derives for `Device`/`Alert`), there is no analogous type-level signal here at all — every resource caller of `httpGet`/`sendWrite`'s `validateResponse`/`paginate`'s `validateArrayResponse` inherits a type that is silently narrower than what successfully parses. No test in `base-resource.test.ts`/`paginate.test.ts` exercises a `null`-valued named field to document (even informally) that this narrowing is deliberate. The doc comment's rationale ("a `T`-typed field being unexpectedly `null` at runtime is exactly the reality response leniency exists to tolerate, not a guarantee this cast invents") describes the runtime behavior accurately but does not make the *type* honest — it is the same trade-off Phase 4 explicitly rejected for `parseLenient` itself. | Either (a) change `validateResponse`/`validateArrayResponse`'s return types to `Lenient<T>`/`Lenient<T>[]` and let Phase 7/8 resource methods declare their real return type honestly (mirroring how `Device`/`Alert` already graft the enum widening rather than hiding it), or (b) if the team deliberately wants resource methods to expose the clean `T` shape despite the null-tolerance gap, escalate that as an explicit, ruled decision (it reverses Phase 4's own fix) and add a permanent test — e.g. a `null`-valued field flowing through `validateResponse` — documenting the accepted gap the way `tests/generated/lenient-type-pin.ts` documents `Lenient<T>` itself, so the trade-off is verified and visible rather than only asserted in a doc comment. |
| typescript-cop-r1-f2 | Low | Open | PublicTypes | `src/schema-overrides/write-bodies.ts:114-158` (`createSiteVariableWriteBodySchema`, `updateSiteVariableWriteBodySchema`, `createAccountVariableWriteBodySchema`, `updateAccountVariableWriteBodySchema`) | Every other write-body schema in this module (`udfWriteBodySchema`→`DeviceUdfInput`, `warrantyWriteBodySchema`→`DeviceWarrantyInput`, `createSiteVariableWriteBodySchema`→`SiteVariableCreateInput`, `createAccountVariableWriteBodySchema`→`AccountVariableCreateInput`, `updateProxyWriteBodySchema`→`SiteProxyInput`) exports a companion input type from this same file, re-exported through `index.ts`, so a Phase 7/8 resource method can type its own input parameter without reaching into `src/generated/**` directly. `updateSiteVariableWriteBodySchema` and `updateAccountVariableWriteBodySchema` are the only two write-body schemas in the module with no companion type export at all, breaking the module's own stated goal of being "the single place a reader checks for what does this write body actually require" — a resource author implementing the update-half of `site-variable-set`/`account-variable-set` has to fall back to importing `VariableUpdateRequest` from `src/generated/types` instead, splitting the type source across two modules for no documented reason. | Export `SiteVariableUpdateInput`/`AccountVariableUpdateInput` type aliases (`z.infer<typeof updateSiteVariableWriteBodySchema>`/`z.infer<typeof updateAccountVariableWriteBodySchema>`) alongside the existing exports, and re-export them from `index.ts`, matching every sibling schema's pattern. |

No other findings: the `httpPost`/`httpPut`/`httpPatch` overload dispatch narrows correctly on the
fixed-length tuple union (`BodilessWriteArgs`/`BodiedWriteArgs`) with no unsound branch; `sendWrite`'s
shared dispatcher is fully typed with no `any`; `coerceSchema` is the plan-pinned, documented escape
hatch (unused this phase); the `Device`/`Alert` `Omit`/`Pick` open-enum graft in `types.ts` compiles
cleanly against the generated types and is exercised by the implementor's scratch check; `pageDetails`'s
strict-cursor/lenient-item split is sound and its `.catchall` is correctly scoped; `z.prettifyError`
carries no wire value into `validateArrayResponse`'s aggregated `warn` `meta`; no floating promises or
missed `await`s anywhere in the primitive/`paginate` async paths.
