# reviser — round 1

Scope: responses to `implementation-auditor-r1.md`'s four Open findings.

| ID | Disposition | Rationale |
|----|-------------|-----------|
| implementation-auditor-r1-f1 | Fixed | `src/schema-overrides/write-bodies.ts` now reconciles **every** body-carrying write operation named by a Phase 5 `WriteOpKey` (verified directly against `spec/openapi.json`: 9 write bodies total, including `device-udf-set`). Two (`site-create`'s `createBody`, `device-job-create`'s `createQuickJobBody`) already carry their genuinely-required fields straight from the spec's own `components.schemas` `required` arrays (`CreateSiteRequest: required:["name"]`, `CreateQuickJobRequest: required:["jobComponent","jobName"]`) — the generator already emits those fields non-optional, so no wrapper logic was needed; they are re-exported (unchanged) as `siteCreateBodySchema`/`deviceJobCreateBodySchema` so this module stays the single place a reader checks every write body's required-ness. The remaining six get a hand-verified wrapper: `createSiteVariableWriteBodySchema`/`createAccountVariableWriteBodySchema` require `name` (the field's own spec description, "Variable name", makes it the one unambiguous load-bearing field for a *create* — a variable can't be referenced without one); `updateSiteVariableWriteBodySchema`/`updateAccountVariableWriteBodySchema`/`updateProxyWriteBodySchema` apply the same "reject an all-omitted body" `.refine` already used for `device-udf-set`, since for an update/settings body no single field is unambiguously "the" required one (I did not guess a specific required subset there, precisely the risk the original phase notes flagged); `warrantyWriteBodySchema` requires `warrantyDate` present but **nullable** (fixing the `.nullable()` gap the phase notes' own §11 had already flagged, since the endpoint doc explicitly allows clearing via `null`). Every new schema has an accompanying test in `tests/unit/schema-overrides/write-bodies.test.ts` (16 new test cases). While enumerating every non-`GET` `(method, path)` pair to make this claim precise, I found `POST /api/v2/site/{siteUid}` ("Updates the site...", body `SiteRequest`/generated `updateBody`) is a **10th** body-carrying write operation the spec declares, but it has **no corresponding key at all** in Phase 5's `WriteOpKey` union (`rate-limits.ts`) — the inverse of that same file's already-flagged `filter-create`/`filter-delete` dead entries. `BaseResource`'s write primitives require a `WriteOpKey`, so this op cannot be reached from any resource until Phase 5's table gains a key for it; documented as a known gap in `write-bodies.ts`'s own module doc rather than silently reconciling a body for an operation no primitive can currently dispatch, and left for Phase 5/7 triage (editing `rate-limits.ts` is out of this phase's scope, per the coexistence rule this phase's own notes already applied to the sibling `filter-create`/`filter-delete` finding). Re-verified: `npm run lint`/`typecheck`/`test`/`build` all pass (311 tests, 0 lint errors). |
| implementation-auditor-r1-f2 | Fixed | Rewrote the R20-invariant test in `tests/unit/client/base-resource.test.ts`: it now parses `{ udf1: "S3CR3T-RAW-WIRE-VALUE" }` (a real string) against a schema expecting `udf1: z.number()`, so the raw wire value genuinely flows into `dropped[].error` via zod's prettified error text — the one path that could smuggle a value into `meta` outside the UDF-key masker's reach. The test now asserts both that the message string never contains the value (as before) **and**, newly, that `JSON.stringify(meta)` does not contain it either, directly exercising the "meta never leaks a raw value via `firstErrors`" invariant instead of asserting on a value (`"S3CR3T-SHAPE-MISMATCH".length`, a number) that never entered the data at all. |
| implementation-auditor-r1-f3 | Fixed | Restored `paginate`'s trailing `params`/`context` to optional (`params?: Record<string, unknown>, context?: string`) in `src/client/resources/base-resource.ts`, exactly matching the plan's pinned `paginate(startPath, arrayKey, itemSchema, params?, context?)` signature. Internally it falls back to the existing `UNKNOWN_CONTEXT` label when `context` is omitted, so the strict-cursor throw and the `validateArrayResponse` diagnostics stay meaningful either way. `paginate` has no bodied/bodiless overload dispatch (unlike `httpPost`/`httpPut`/`httpPatch`, whose required `context` is separately justified by fixed argument-tail-length dispatch — that justification, and only that one, still applies to the `http*` primitives), so there was no actual reason for `paginate` to diverge from the pinned optional signature. Added a test (`tests/unit/client/paginate.test.ts`, "accepts omitted trailing params/context") proving a call with only the first three arguments still compiles and walks correctly. |
| implementation-auditor-r1-f4 | Fixed | `validateArrayResponse` (`src/client/resources/base-resource.ts`) now branches on `Array.isArray(data)`: a genuinely-empty array (`[]`) still emits nothing (a legitimate empty page), but `data` being `undefined`/`null`/non-array emits its own single, distinct `warn` (`"response array field was not an array"`, `meta: { context, receivedType }`) rather than silently returning `[]` with no diagnostic. Since `paginate` calls `validateArrayResponse` with `data?.[arrayKey]`, a wrong/misnamed `arrayKey` or a schema drift that drops the array field now surfaces instead of being indistinguishable from a legitimately empty page. Added three tests in `base-resource.test.ts` (non-array object, `undefined`, and genuinely-empty-array-emits-nothing) and one in `paginate.test.ts` (a page with no `devices` key at all now warns instead of silently yielding zero items). |

All four fixes were verified together: `npm run lint` (0 errors, 11 pre-existing warnings in the untouched old surface, unchanged from baseline), `npm run typecheck`, `npm test` (311 tests passing, up from 293 — all new tests included), and `npm run build` all green.
