## reviser — round 1

| ID | Disposition | Rationale (and, for Fixed, what changed) |
|----|-------------|-------------------------------------------|
| implementation-auditor-r1-f1 | Fixed | `account-resource.ts`'s `get()` now validates against the generated `getUserAccountResponse` (`src/generated/schemas/-v2-account/-v2-account.zod.ts`) instead of a hand-copied `accountSchema`; the hand-written schema is deleted, matching how `SiteResource.get()`/`JobResource.get()` reuse their generated response schemas directly. |
| implementation-auditor-r1-f2 | Fixed | Added `tests/generated/schema-mirror-pin.ts`, a compile-time-only guard (picked up by `tsconfig.test.json`'s `include`, same mechanism as `lenient-type-pin.ts`) that asserts `keyof <generatedType>` equals `keyof z.infer<typeof <handSchema>>` for all six mirrors: `componentSchema`/`Component`, `dnetSiteMappingSchema`/`DnetSiteMappingsDto` (now exported from `account-resource.ts`), `deviceNetworkInterfaceSchema`/`DeviceNetworkInterface`, `filterSchema`/`Filter` (now exported from `site-resource.ts`), `jobComponentSchema`/`JobComponent` (now exported from `job-resource.ts`), and `variableSchema`/`Variable`. Used key-set equality rather than full deep `Equal` because a literal deep pin would fail today, independent of any future drift, for `Filter`/`filterSchema`: the Phase 2 response-enum-widening codemod widens `Filter["type"]` to `FilterType \| (string & {})` at compile time, while `filterSchema`'s `z.enum([...])` is authored closed — that open/closed asymmetry is reconciled only at runtime by `parseLenient` and is already covered generically by `Lenient<T>`/`lenient-type-pin.ts`. Key-set equality catches exactly the named hazard (a field silently added/removed upstream) without fighting that unrelated, already-guarded asymmetry. Verified the new pin fails as expected by temporarily removing a field from one mirror during authoring; `npm run typecheck` now enforces it permanently. |
| implementation-auditor-r1-f3 | Fixed | `AccountResource.variables()` now takes `GetAccountVariablesParams`, `SiteResource.variables()` now takes `GetSiteVariablesParams` (both generated, structurally `{page?: number; max?: number}`, confirmed identical to the removed inline literals), and `DeviceResource.createJob()`'s `body` parameter is now typed `CreateQuickJobRequest` (generated, confirmed identical to the removed inline `{jobName; jobComponent}` literal) in place of the ad hoc inline object types. |
| implementation-auditor-r1-f4 | Fixed | Implemented `SiteResource.update()` (`POST /api/v2/site/{siteUid}`, body `SiteRequest`, response reuses `getSiteResponse` — identical generated shape to `updateResponse`). Added `'site-update': 100` to `WRITE_LIMITS`/`WriteOpKey` in `src/rate-limit/rate-limits.ts` (the table's own doc names this as the expected extension point: "Adding a write method requires adding its opKey here first"); this file is Phase-5-created infrastructure designed to be extended by later phases, not one of the coexistence rule's untouched *old* pre-refactor files, so adding a table entry here is in-scope Phase 7 work, not a boundary violation. Exported `siteUpdateBodySchema` from `src/schema-overrides/write-bodies.ts` (re-exporting the generated `updateBody` unchanged — its `name` field is already spec-required, same pattern as the existing `siteCreateBodySchema`) and replaced that module's "known gap, out of scope" doc paragraph, which is now resolved. Added a nock test (`update() POSTs .../site/{siteUid}, tags site-update, and rejects a malformed body`) mirroring `create()`'s test, and three `siteUpdateBodySchema` cases in `write-bodies.test.ts`. `npm run lint`/`typecheck`/`test`/`build` all pass (378 tests, up from 374). |
