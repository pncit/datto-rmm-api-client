## 2. Phase Intent (Interpreted)

Swap the build/test toolchain from `tsc`+`jest` to `tsup`+`vitest`(+`nock`, unused yet), and wire
(but do not exercise) Orval codegen, so that every later phase builds on the target toolchain from
day one. The existing three-method, `Result<T>`-returning surface must still compile and its tests
must still pass — now under vitest — proving the runner swap is behavior-neutral. `npm run
generate` is present in `package.json` but its constituent scripts (`scripts/patch-spec.mjs`,
`scripts/dedupe-generated-index.mjs`, `scripts/widen-response-enums.mjs`) don't exist yet and no
spec is committed yet, so this phase only proves the `orval` binary is installed and runnable
(`npx orval --help`), not a working generate pipeline.

## 4. Implementation Summary

**Dependencies & scripts.** `package.json` now matches the plan's pinned versions
(`orval@^7`, `tsup@^8`, `vitest@^4`, `nock@^14`, `@vitest/coverage-v8@^4`, `@types/node@^22` — the
Node-20-floor pin, deliberately not `fuze-api`'s `^26`). `zod@^4.0.5` and `axios@^1.10.0` are
untouched runtime dependencies. Scripts were rewritten exactly per the plan's Step 2: `generate`
chains `patch-spec.mjs → orval → dedupe-generated-index.mjs → widen-response-enums.mjs` (none of
those three scripts exist yet — they're Phase 2 deliverables — so `npm run generate` is wired but
intentionally not invoked by this phase's gate); `generate:raw` runs bare `orval`; `build` is
`tsup`; `test`/`test:watch` are `vitest run`/`vitest`; `typecheck` unchanged; `clean` unchanged;
`prepublishOnly` drops the old `npm run clean` step since `tsup`'s `clean: true` already clears
`dist` on every build.

**Codegen/build/test configs.** `orval.config.ts` defines the two targets (`datto` for axios
types + a git-ignored endpoints stub, `dattoZod` for zod schemas) exactly per the plan's snippet,
reading from `spec/openapi.patched.json` (overridable via `DATTO_OPENAPI_SPEC`) — that file does
not exist yet, which is fine since this phase never invokes Orval against it. `tsup.config.ts` is
single-entry (`index: 'src/index.ts'`), ESM-only, `dts: true` — deliberately *not* copying
`fuze-api`'s two-entry (`index` + `index.browser`) shape, since browser support is an explicit
Non-Goal for this client (Non-Goals, design). `vitest.config.ts` sets `globals: true`,
`environment: 'node'`, `include: ['tests/**/*.test.ts', 'src/**/*.test.ts']` (the existing tests
live under the latter), coverage excludes `src/generated/**` and `src/index.ts`, and the `@` →
`./src` alias. No `tests/setup.ts` was added: nothing in this phase's tests needs global setup
(no real network is touched — the converted tests use the same hand-rolled `MockAxios` stub as
before), matching the plan's "otherwise omit" guidance.

**tsconfig.** Added `"baseUrl": "."` and `"paths": { "@/*": ["./src/*"] }`, and switched
`moduleResolution` from `Node` to `Bundler`. `module` was already `ESNext`, satisfying the
`Bundler`+`ESNext` pairing requirement without needing a separate edit.

**Test conversion.** All four `src/__tests__/*.test.ts` files were converted mechanically:
`jest.fn()` → `vi.fn()`, `jest.spyOn` → `vi.spyOn`, the `(x as jest.Mock)` cast pattern → `(x as
Mock)` using vitest's exported `Mock` type, and every test file now imports `describe`/`test`/
`expect`/`vi`/`Mock` explicitly from `'vitest'` (matching `fuze-api`'s own test convention) rather
than relying solely on `vitest.config.ts`'s `globals: true` — this keeps the files self-documenting
and avoids depending on ESLint's `no-undef` being off for correctness. Two files
(`deviceSchema.test.ts`, `devicesMethod.test.ts`) used bare `__dirname` to locate fixture JSON;
under this package's `"type": "module"` (ESM), `__dirname` is not a real ESM global, so both were
switched to the standard ESM idiom
`path.dirname(fileURLToPath(import.meta.url))`. No test assertions, fixtures, or business logic
changed — only the harness.

**`.gitignore`.** Added `spec/openapi.patched.json` (the transient patch-step output) and
`src/generated/endpoints/` (the git-ignored-by-design Orval endpoints stub — never imported, per
the plan's "no dead second source of truth for paths" resolution) with a comment noting that
`src/generated/types/` and `src/generated/schemas/` are *not* ignored and will be committed in
Phase 2.

**eslint.config.js.** The test-file `globals` block referenced Jest's implicit globals (including
`jest: 'readonly'`); updated to `vi: 'readonly'` to match the new runner, with a comment explaining
these declarations are a belt-and-suspenders complement to `vitest.config.ts`'s `globals: true`
(the actual test files import everything explicitly, so this mainly guards against a future file
that doesn't).

## 5. Deviations From Plan (If Any)

- **`tsup.config.ts` single-entry only, no `index.browser`.** The plan's Step 3 explicitly calls
  for "Single entry; no browser build — browser is a Non-Goal," which is what was implemented;
  flagging this only because `fuze-api`'s own `tsup.config.ts` (referenced as the copy source) has
  two entries — the deviation from the *literal source file* is plan-directed, not a judgment call.
- **`__dirname` via `fileURLToPath(import.meta.url)` instead of a bare `__dirname` global.** The
  plan doesn't mention this explicitly (its own fuze-api precedent doesn't load fixtures from disk
  by path). This was a necessary fix, not a stylistic choice: the package is `"type": "module"`, so
  a bare `__dirname` is `undefined` under real Node ESM execution; the previous jest setup only
  worked because `ts-jest`'s ESM preset shims it. Vitest's Vite-based transform does not provide
  that shim, so the tests would fail to locate fixtures without this change. Used the standard ESM
  idiom rather than reaching for a non-standard runtime global.
- **eslint.config.js test-globals update.** Not an explicit plan step, but a direct consequence of
  retiring jest (Non-Negotiable Rule: fix build/test breakage the current work causes) — the `jest`
  global declaration is now stale and would silently permit a bare `jest.fn()` typo to lint clean
  post-migration; replaced with the `vi` equivalent. Labeled here as a **Phase 1 necessity**, kept
  minimal (one key swapped, one clarifying comment).

No other deviations.

## 6. Ambiguities & Decisions

- **`vitest.config.ts` `setupFiles`.** The plan says to add `tests/setup.ts` "if any global test
  setup is needed (nock disabling real net); otherwise omit." This phase's tests don't touch `nock`
  or real network (they use the pre-existing `MockAxios` stub), so `setupFiles` was omitted
  entirely, per the plan's own fallback instruction.
- **`@types/node` version.** Pinned to `^22.0.0` per the plan's explicit Node-20-floor rationale
  (not `fuze-api`'s `^26`), verified against `engines.node: >=20.0.0` (unchanged this phase).
- **Explicit vitest imports vs. relying on `globals: true`.** The plan allows either ("replace jest
  globals with vitest imports (or rely on `globals: true`)"). Chose explicit imports (matching
  every `fuze-api` test file inspected) for self-documentation and IDE-goto-definition; `globals:
  true` is still set in `vitest.config.ts` per the plan's template, so a future file's omission of
  imports still works.
