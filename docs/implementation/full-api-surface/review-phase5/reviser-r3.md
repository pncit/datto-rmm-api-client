## reviser — round 3

| ID | Disposition | Rationale (and, for Fixed, what changed) |
|----|-------------|-------------------------------------------|
| engineer-r2-f1 | Fixed | `performRefresh`'s malformed-token-response branch (`src/auth/auth-manager.ts`) no longer attaches the raw grant body as `response`. A 200 grant reply that fails `tokenResponseSchema` (e.g. a valid `access_token` with a bad `expires_in`) now throws `DattoApiError` with `statusCode`/`cause` (the zod error) only — `response` is omitted (`undefined`), so a live bearer token can no longer reach the public `DattoApiError.response` field a consumer might log. Extended the existing "non-numeric expires_in" test in `tests/unit/auth/auth-manager.test.ts` to use a realistic token value and assert both `error.response` is `undefined` and the token never appears in the error's own-property JSON serialization. |
| project-lead-r2-f1 | Fixed | `handleResponseError` (`src/http/http-client.ts`) now guards at the top with `if (!axios.isAxiosError(error)) throw error;` before touching `.response`/`.config`, and its parameter (and the response interceptor's rejection-handler type) changed from `AxiosError` to `unknown` to reflect axios's actual `(error: any) => any` interceptor typing. An already-typed error thrown by an upstream request interceptor on the same instance (e.g. a future `AuthManager.attachTo` wiring whose `getToken()` throws `DattoApiError` on a failed/malformed grant) now propagates unchanged instead of being lossily reconstructed via `DattoApiError.fromAxiosError` with a discarded real `statusCode`/`response`/`requestId`. Added a test in `tests/unit/http/http-client.test.ts` that registers a request interceptor throwing a `DattoApiError` on a `createHttpClient`-built instance and asserts the identical error object (same `statusCode`/`response`) reaches the caller. |
