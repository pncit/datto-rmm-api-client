## Findings

| ID | Severity | Status | Category | Location | Finding | Recommendation |
|----|----------|--------|----------|----------|---------|----------------|
| project-lead-r1-f1 | Medium | Open | BehaviorIntent | `src/http/http-client.ts` `build403Error`/`buildRateLimitError` (l.460-482); cf. `src/errors/datto-api-error.ts` `extractRequestId`/`fromAxiosError` | `DattoApiError`'s own contract (Phase 3, exercised by `tests/unit/errors/datto-api-error.test.ts`) is "captures the HTTP status, the raw response body, a server-supplied request id (when present)". `fromAxiosError` honors that by extracting `requestId` from conventional headers. But this phase's two *direct*-construction paths — `build403Error` (both `ip-block` and `forbidden`) and `buildRateLimitError` (429 bounded-out/exhausted) — never call anything equivalent; `extractRequestId` isn't even exported from `src/errors/index.ts` for them to reuse. Every 403 and every 429-that-gives-up thrown by this phase's transport therefore has `requestId: undefined` even when the response carries a conventional request-id header — losing the one field a consumer would use to correlate exactly these two failure modes (rate-limit block, IP block) with Datto support. No test in `tests/unit/http/http-client.test.ts` asserts `requestId` on any 403/429 case, so the gap wasn't caught. | Export `extractRequestId` (or an equivalent) from `src/errors`, and pass `requestId: extractRequestId(error.response?.headers)` into both `build403Error`'s and `buildRateLimitError`'s `DattoApiError` construction; add an assertion in at least one 403 and one 429-exhaustion test that a request-id response header round-trips into `error.requestId`. |
| project-lead-r1-f2 | High | Open | Risk | `src/errors/datto-api-error.ts` `fromAxiosError` (sets `cause: err`), consumed by `src/http/http-client.ts` (`build403Error`, `buildRateLimitError`, and the `fromAxiosError` fallback) and `src/auth/auth-manager.ts` `refreshToken`'s catch block | Every `DattoApiError` this phase's transport throws carries the full, unredacted `AxiosError` as `Error.cause`. For a v2 API failure that includes `error.cause.config.headers.Authorization: "Bearer <token>"` — the live access token. For an OAuth2 grant/refresh failure (`auth-manager.ts`'s catch, which rethrows via `DattoApiError.fromAxiosError`), `error.cause.config.data` is the URL-encoded grant body `grant_type=password&username=<apiKey>&password=<apiSecret>` — the caller's raw account credentials, in cleartext. `Error.cause` is a non-enumerable own property, so a plain `JSON.stringify(err)` doesn't surface it, but Node's own `console.error`/`util.inspect` (and most crash-reporting/log-aggregation integrations, which specifically walk `Error.cause` chains) print it — meaning a bare `console.error(err)` in any consumer's catch block, or an APM/error-tracking integration, leaks the account's live Bearer token or its raw `apiKey`/`apiSecret` into logs. The design places explicit weight on not leaking secrets (R20's UDF masking is the analogous concern for response data); this phase introduces the first place real credentials/tokens flow through the transport, and nothing scrubs the attached cause before it's thrown. The phase's own Security & Best-Practices Review (§8) asserts credentials are "never placed in an error message or thrown value," which is true of the *message* but not of the *cause* attached to that same thrown value. | Before attaching `err` as `cause` (in `DattoApiError.fromAxiosError`, or specifically at the two Phase 5 call sites that matter most — `build403Error`/`buildRateLimitError` and `AuthManager.refreshToken`'s catch), strip/redact `config.headers.Authorization`, `config.auth`, and `config.data` from the `AxiosError` (or its `config`) before it becomes `cause`, or omit `cause` entirely for the grant/refresh failure path and instead carry only the already-safe `statusCode`/`response`. Add a test asserting a thrown `DattoApiError` from a failed grant does not expose `apiSecret`/`apiKey` anywhere reachable via `error.cause`. |
| project-lead-r1-f3 | Medium | Open | Risk | `src/http/http-client.ts` `isRateLimitBlock` (l.432-451); `tests/unit/http/http-client.test.ts` | `isRateLimitBlock` classifies via two independent OR'd signals: a `Retry-After` header, or a rate-limit/block wording match in the body. Only the body-wording branch is exercised by a test (the "classified ip-block" test's fixture has no `Retry-After` header, relying solely on the message match). The `Retry-After`-header-alone branch — the *first* check in the function, and per its own doc "the server's own explicit throttling signal" — has zero test coverage anywhere in the suite (the standalone `isRateLimitBlock` describe block only covers the `undefined`-response case). This is exactly the class of classification logic Phase 5's Non-Goal ("no auto-recovery from a 403 IP-block") depends on a consumer correctly distinguishing, so an unverified branch here is a real gap in the safety net around a stateful, account-wide penalty. | Add a test with a 403 whose body does **not** match the rate-limit wording pattern but whose response carries a `Retry-After` header, asserting `isRateLimitBlock` (and the end-to-end `DattoApiError.code`) still resolves to `'ip-block'` via the header alone. |
