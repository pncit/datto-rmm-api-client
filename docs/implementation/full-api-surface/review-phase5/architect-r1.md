## architect — round 1

First architect turn on Phase 5 (rate limiting, HTTP transport, throwing auth). Prior turns in this
directory are the implementation-auditor's (r1/r2) and the reviser's (r1); those two auditor findings
(`implementation-auditor-r1-f1`/`f2`) were test-coverage gaps, ratified `Closed` by the auditor in
r2, and I do not re-open them. My review is a fresh architectural pass over the untracked Phase 5 tree
(`src/rate-limit/`, `src/http/`, `src/auth/`) plus the one-line `tsconfig.test.json` change, scoped
via `git diff` against `main`.

The module boundaries and dependency direction are sound: `defaults.ts` is correctly layer-neutral,
the rate-limit table is co-located with its limiter, the ambient axios augmentation is kept out of the
published-types graph, and transport isolation of the grant round-trip (bare `grantClient`) is
structurally real. `acquire`'s throttle-not-reject model and check-then-record concurrency are correct.
My findings below are boundary/data-handling gaps the plan's own text calls for (or that a transport
meant to back many concurrent resource calls needs) but that the implementation does not yet deliver.

### Findings

| ID | Severity | Status | Category | Location | Finding | Recommendation / update |
|----|----------|--------|----------|----------|---------|-------------------------|
| architect-r1-f1 | Medium | Open | Security / Data handling | `src/auth/auth-manager.ts` `refreshToken` catch, l.123-131 (`DattoApiError.fromAxiosError(err)` / `cause: err`); flows into `DattoApiError` `cause` (`src/errors/datto-api-error.ts` l.169-174, `base-error.ts` l.6-7) | The grant body carries `username`/`password` (= the caller's `apiKey`/`apiSecret`, `auth-manager.ts` l.104-108). On a failed grant the raw `AxiosError` is wrapped as `cause` on the thrown `DattoApiError`. `AxiosError.toJSON()`/serialization includes `config` (with `config.data` — the URL-encoded body — and `config.headers`), so any consumer that logs or `JSON.stringify`s the error's `cause` chain leaks the API secret in plaintext. The notes' §8 claim that credentials are "never placed in an error message or thrown value" is incorrect for the cause chain. The same vector exposes the `Authorization: Bearer <token>` header for any shared-instance request failure wrapped by `fromAxiosError`. | On the auth failure path (and in `fromAxiosError`), scrub before wrapping: strip/redact `err.config.data` and `err.config.headers.Authorization` (or set `cause` to a sanitized copy rather than the raw `AxiosError`) so no serialization of the thrown error can surface `apiSecret`/`password`/bearer token. |
| architect-r1-f2 | Medium | Open | Data flow / Boundary mapping | `src/http/http-client.ts` `build403Error` l.190-203 (only `response: error.response?.data`; no `retryAfterMs`, no headers) | Plan Step 3(c) requires a 403 to surface "the raw `response` body **and headers** so a consumer can disambiguate," and an `ip-block` 403 is classified precisely *because* it carries a `Retry-After` header (`isRateLimitBlock`, l.160). Yet `build403Error` drops that header entirely: `DattoApiError` has no headers carrier and `retryAfterMs` is left unset, so a consumer that learns `code:'ip-block'` cannot recover *how long* the block lasts — the one actionable datum on an IP block is discarded even though the code just read it. | For the `ip-block` branch, parse the 403's `Retry-After` (reuse `parseRetryAfterMs(readHeader(error.response?.headers, "Retry-After"))`) and pass it as `retryAfterMs` so the block duration survives on the thrown error, matching the plan's "disambiguate" intent within `DattoApiError`'s existing fields. |
| architect-r1-f3 | Medium | Open | Concurrency / Boundary | `src/auth/auth-manager.ts` `getToken` l.80-86 + `refreshToken` l.103-140 | No single-flight coalescing on refresh. `attachTo` installs the auth interceptor on the *shared* instance that Phase 6+ resources drive concurrently; a burst of N requests hitting an empty/stale cache each independently invoke `refreshToken()`, firing N simultaneous OAuth2 grant round-trips (each overwriting the prior's cached token) — self-inflicted load on the auth endpoint that can itself trip Datto's limits, precisely at client startup. This is inherent to a transport built for concurrent resource calls, not a corner case; the notes acknowledge it (§11) but defer it. | Coalesce in-flight refreshes: cache the pending `refreshToken()` promise on the instance and have concurrent `getToken()` calls await the same promise, clearing it on settle, so a burst produces exactly one grant round-trip. |
| architect-r1-f4 | Medium | Open | Resilience / Hot path | `src/http/http-client.ts` `createHttpClient` `axios.create` l.270-276; `auth-manager.ts` `grantClient` l.55-58 | Neither the shared instance nor `grantClient` sets a request `timeout`, and `HttpClientConfig`/`AuthManagerConfig` expose none. With no timeout a stalled/half-open connection hangs the request indefinitely; because the retry logic keys entirely off a surfaced error (`isNetworkError = error.response === undefined`, l.249), a hang never errors and the network-retry path never fires — the whole retry/backoff design is defeated by the one failure mode it most needs to cover. | Set a sensible default `timeout` on both `axios.create` calls (a `DEFAULT_TIMEOUT_MS` in `src/defaults.ts`, mirroring `DEFAULT_RETRY`), so a stalled request converts to a retryable/throwable error instead of hanging. |
| architect-r1-f5 | Low | Open | Boundary / Lifecycle seam | `src/http/http-client.ts` `handleResponseError` l.212-260 (401 falls through to `fromAxiosError`); `auth-manager.ts` `invalidate` l.143-145 (no caller) | A 401 (token revoked server-side, or expired before the 25%-remaining proactive refresh window) is treated as a generic non-retryable error; nothing invalidates the cached token or re-authenticates, so a stale token yields a permanent hard failure until the *original* TTL elapses. `AuthManager.invalidate()` exists solely for this and currently has no caller anywhere, and no plan/design text assigns ownership of the 401→invalidate seam. The seam is undefined right where the two modules meet. | Either wire a 401 branch in `handleResponseError` that calls `authManager.invalidate()` and retries once, or explicitly document (module doc + plan note) that 401-recovery is the client/`BaseResource` layer's responsibility and give `invalidate()` its intended caller there — don't leave a dead recovery method and an unowned failure mode. |
| architect-r1-f6 | Low | Open | Maintainability / DRY | `src/http/http-client.ts` `sleep` l.81-85 & `isRecord` l.122-124; `src/rate-limit/rate-limiter.ts` `sleep` l.34-38; `src/errors/datto-api-error.ts` `isRecord` l.53-55 | `sleep` and `isRecord` are each hand-duplicated across Phase 5 modules, and header-reading logic is re-implemented three ways (`readHeader` in http-client vs `extractRequestId`/`firstNonEmptyString` in errors). These are exactly the small transport utilities that drift (e.g. one `sleep` gaining a cancel token, one `isRecord` tightening) as the surface grows across Phases 6-8. | Extract the shared primitives (`sleep`, `isRecord`, case-insensitive header read) into one small internal util module (e.g. `src/internal/`) and import from both transport modules, so there is a single definition per primitive. |
