| project-lead-r2-f1 | High | Open | Risk | `src/http/http-client.ts` `handleResponseError` (l.238-318), consumed by `createHttpClient`'s response interceptor (l.346-356); composition documented in the module doc (l.37-43: "`AuthManager` … attaches its own request interceptor onto **the instance this module returns**") and in `AuthManager.attachTo` (`src/auth/auth-manager.ts` l.76-89) | `handleResponseError` is typed to receive an `AxiosError` and, once a `config` is present, unconditionally treats status codes and, on any unhandled path, calls `DattoApiError.fromAxiosError(error)` — assuming `error` is always a real `AxiosError`. That assumption is false for the module's own documented architecture: `AuthManager.attachTo(instance)` (intended to be called on **this same shared instance**, per this module's doc and the plan's Phase 5 Step 4 "expose the token via a request interceptor on the shared instance") installs a **request** interceptor whose `getToken()`/`refreshToken()` call **throws a `DattoApiError`** on a failed/malformed grant. I verified empirically (a standalone axios repro script) that a rejection thrown from a request interceptor on an instance is delivered to that **same instance's response-interceptor rejected handler** — i.e. to `handleResponseError` — not just to the final caller. So the very first request made on a misconfigured client (bad `apiKey`/`apiSecret` — an ordinary, common failure, not an exotic race) throws a `DattoApiError` from the auth interceptor, which `handleResponseError` receives as its `error` parameter: `error.response?.status` reads `.status` off `DattoApiError.response` (the raw grant-failure body, not a nested `{status}` shape) and is normally `undefined`, `error.config` is `undefined` (no such field on `DattoApiError`), so the `!config` branch fires and rethrows via `DattoApiError.fromAxiosError(error)` — called on a non-`AxiosError`. That reconstructs a **new** `DattoApiError` with `statusCode: 0` (discarding the real status, e.g. `503`/`401` from the grant failure), `response: err.response?.data` (almost always `undefined`, discarding the real grant-failure body), `requestId: undefined` (discarding a real one if present), and a `cause` rebuilt from `sanitizeAxiosErrorCause(err)` reading `err.config?.method`/`err.config?.url` (both `undefined` on a `DattoApiError`) — silently corrupting/discarding the original, correctly-typed error's real diagnostic fields. This is untested: `AuthManager.attachTo` is only ever exercised in tests against a bare `axios.create()` instance (`tests/unit/auth/auth-manager.test.ts`), never against `createHttpClient`'s output, so this composition — which the module's own doc states as the intended wiring — has zero coverage and a live, will-reproduce-in-Phase-7 defect. | Guard at the top of `handleResponseError` (before the 403 check): `if (!axios.isAxiosError(error)) throw error;` (the module already imports the `axios` default) so an already-well-typed error thrown by an upstream request interceptor on the same instance (`AuthManager`'s Bearer-attaching interceptor, or any future one) propagates unchanged instead of being lossily re-wrapped through the `AxiosError`-specific 403/429/`fromAxiosError` paths. Add a test that registers a request interceptor which throws a `DattoApiError` on an instance built via `createHttpClient`, and asserts the identical error (same `statusCode`/`response`/`cause`) is what the caller receives. |
