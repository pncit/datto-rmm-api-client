## engineer — round 1

Code-review of Phase 9 (fixtures, sanitizer, reconciled-schema validation). Scope taken from
`git diff` against the default branch, narrowed to the phase's additive deliverables:
`scripts/sanitize-fixtures.mjs`, `tests/unit/scripts/sanitize-fixtures.test.ts`,
`tests/integration/fixtures.test.ts`, and the new `tests/fixtures/*.json`. No `src/` change, as the
notes claim. The reviser has already landed the CLI/`main()` tests the auditor asked for, so that
gap is not re-raised.

The deliverables are well-structured, reuse existing primitives (`BaseResource`, `enumFieldPaths`,
`OVERRIDE_ENTITIES`, `withUdfMasking`) rather than re-deriving them, and the tests bind to real
behavior. Findings below are maintainability/test-fidelity issues; none blocks the build.

## Findings

| ID | Severity | Status | Category | Location | Finding | Recommendation / update |
|----|----------|--------|----------|----------|---------|-------------------------|
| engineer-r1-f1 | Medium | Open | Complexity | tests/integration/fixtures.test.ts (`WIDENED_FIELDS completeness guard`) | The completeness guard only asserts one direction — every enum field's top-level ⊆ `widenedFields`. The reverse (every `widenedFields` entry actually corresponds to an enum-bearing field) is never checked, yet `types.ts`' own doc-comment claims the graft and constant "cannot desync" / "cover the same field set." A stale or mistaken `WIDENED_FIELDS` entry (a top-level with no enum) would `Omit` the reconciled field and re-`Pick` the generated one — silently undoing that field's reconciliation (e.g. a nullability widening) — and this gate would pass. The claimed invariant is only half-enforced. | Add a reverse assertion in the same loop: for each `entry.widenedFields`, assert it is the top-level of at least one path returned by `enumFieldPaths(entry.schema)` (or is an explicitly-listed, documented no-enum reconciled field). With today's constants (`deviceClass`/`antivirus`/`patchManagement`, `priority`/`responseActions`) this passes and closes the desync-in-the-other-direction hole. |
| engineer-r1-f2 | Low | Open | Logging | tests/integration/fixtures.test.ts (`UDF masking never emits a raw fixture UDF value`) | The `validator.validateOne(data, deviceSchema, …)` call in the R20 masking test is inert for its stated purpose: `device-rmmnetworkdevice.json` validates cleanly, so `validateResponse`/`parseLenient` emits no log at all through the wired `maskedLogger`. The only assertion-bearing log is the hand-rolled `maskedLogger.debug(...)`. Notes §4.5 claim the fixture is "validated through the masked-logger-wired path and logged directly" — but the validation half proves nothing (a clean parse never logs), so R20 is only proven against a manual `debug` call, not the actual validation code path. | Either drop the inert `validateOne` call (keep only the direct `maskedLogger.debug` proof and adjust the comment), or drive a validation path that genuinely logs UDF-bearing `meta` — e.g. feed a malformed-item page whose surviving/dropped item carries a `udf` value so `validateArrayResponse`'s `warn` diagnostic actually flows through the masked logger. |
| engineer-r1-f3 | Low | Open | DRY | tests/integration/fixtures.test.ts (`ALERT_CONTEXT_FIXTURES` blocks) | `ALERT_CONTEXT_FIXTURES` is iterated twice: `it.each(...)` asserts only `not.toThrow`, and a separate single `it(...)` re-loops the same six fixtures in a manual `for` asserting field preservation. The `not.toThrow` cases are subsumed by the preservation cases (a preserved field implies a successful parse), and the manual `for` collapses all six into one test — the first failure hides the rest and the failure message doesn't name the fixture. | Fold both into one `it.each(ALERT_CONTEXT_FIXTURES)` that validates and asserts each `@class`-specific field survives, restoring per-fixture isolation and removing the duplicate loop. |
| engineer-r1-f4 | Low | Open | ErrorHandling | scripts/sanitize-fixtures.mjs (`main`) | The script's stated safety property (notes §6: "explicit input/output paths, never overwriting the raw capture in place… safer than an implicit in-place rewrite") is not enforced. `main()` never checks that `outputPath` differs from `inputPath`; invoking `sanitize-fixtures.mjs raw.json raw.json` overwrites the raw capture in place, destroying the un-sanitized source the maintainer is told this script will preserve. The CLI test only exercises the happy path with distinct paths, so the guarantee is asserted in prose but nowhere in code. | Add a guard at the top of `main()`: `if (resolve(inputPath) === resolve(outputPath)) { console.error("sanitize-fixtures: output path must differ from input path"); process.exitCode = 1; return; }` (import `resolve` from `node:path`), and add a CLI test for the same-path branch. |
