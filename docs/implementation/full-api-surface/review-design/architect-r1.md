## architect — round 1

First architect review of this design (the prior turns in this dir are the `design-auditor`
lineage, which converged on requirements-traceability and codegen feasibility; I did not re-raise any
of those). Design Review Mode / restraint posture: I anchored to the stated Goals & Non-Goals, and I
favor findings that **tighten, remove, or clarify** over ones that add scope. Two of my findings are
`Defer:` (correctly-excluded concerns recorded for a follow-up issue, not to be folded in).

The design is strong on problem framing (axis a) and on architectural approach + alternatives
(axis b) — Decisions 1–5 each carry credible rejected alternatives, and the Non-Goals are crisp. My
findings cluster on axes (c) data flow and (d) public API / integration points: two of the
design's most load-bearing guarantees (the R20 masking guarantee and the per-operation rate-limit
contract) are described as *behavior* without pinning down the *boundary* that structurally enforces
them, which leaves them under-plannable and dependent on per-call-site discipline. The rest are
brevity/consistency tightenings.

| architect-r1-f1 | Medium | Open | Architecture | Dual-layer rate limiter Key Concept (l.299–308); R11; Decision 3 | The per-operation write windows are keyed by `operationWriteStatus` names with concrete limit *values* (`device-udf-set`→600, others→100), but the design never says **where those values live or how they are seeded**. Current State notes `system/request_rate` "is queryable" and Future Considerations *defers* adaptive throttling — implying the values are a static hand-maintained table, yet no such table's home is named, nor its init-time behavior (the limiter must have limits before the first request, i.e. before any `requestRate()` call). This is the one place the limiter is genuinely un-plannable: a Planner cannot build it without this decision. | Name the source of truth for the opKey→limit map (a committed static table module vs. a lazy read of `system/request_rate` at client init), and state the default/fallback limit applied to a write opKey with no explicit entry. |
| architect-r1-f2 | Medium | Open | Security | UDF log-masking Key Concept (l.294–298); R20; Success Criteria (l.495–497) | R20's guarantee ("no UDF value in cleartext, ever") is implemented by "a masking utility applied to any payload before it reaches the logger … invoked **wherever** a UDF-bearing payload could be logged." That is guarantee-by-call-site-discipline: any future log site (a new resource, a new diagnostic) that forgets to call the masker silently breaks the guarantee, and the success criterion exercises only one path. A security invariant should be structural, not remembered. | Enforce masking at the **single logger boundary** — wrap the injected `DattoLogger` in a masking decorator through which all client log calls flow — so no call site can emit an unmasked `udf*` value. Then no per-site discipline is required and the guarantee holds by construction. |
| architect-r1-f3 | Medium | Open | Security | Risks table, final row (l.532); R17 | The highest-rated risk (fixtures carrying live BitLocker keys / credentials — **High likelihood / High impact**) is mitigated only by "a **documented** sanitization step gates fixture capture." A documented manual step is the weakest possible control for the worst-rated risk; one skipped step commits a real secret to git history irreversibly. | Make the gate mechanical, not procedural: a committed sanitization/scan script (or pre-commit / CI check over `spec`+fixture paths) that fails when an unsanitized `udf*`/credential-shaped value is present, so the "never commit raw sweep output" guarantee does not depend on remembering the step. |
| architect-r1-f4 | Low | Open | Architecture | R3 (l.76); schema-override concept (l.248–250); BaseResource concept (l.276–284); R5 (l.78) + leniency concept (l.255–272) | Two mechanisms are restated near-verbatim across three locations each, inflating the document against the design-review brevity bar. Tightening only; no behavior change. |
| architect-r1-f5 | Low | Open | PublicAPI | Public surface namespace list (l.444) vs. R1 (l.74) / R2 | The top-level namespaces frozen at `1.0.0` mix pluralization inconsistently: `sites`/`devices`/`alerts`/`jobs`/`filters`/`users`/`activityLogs` (plural) alongside `account`/`audit`/`system` (singular) — while R1 lists the same groups all-singular. `account`/`system` read as singletons, but `audit` vs `audits` (a collection group) is an arbitrary split. Inconsistent names on a surface that ships breaking and cannot be re-cut without a `2.0.0`. | Fix a single pluralization rule before the freeze (e.g. plural for collection groups, singular only for genuine singletons) and reconcile R1/R2/Public-surface to it, so `audit` isn't the lone unexplained singular collection. |
| architect-r1-f6 | Low | Open | Architecture | Vision (l.40–41); Decision 1 rationale (l.318–322) | `Defer:` The convergence goal ("share one mental model / logger contract / validation patterns") is realized by **copying** `fuze-api`'s `parseLenient` and `BaseResource` ("port … rather than invent"), producing two divergent copies of the same components. That re-creates the exact "diverge by historical accident" drift the Problem Statement condemns — the copies will skew on the next fix to either side. This is out of scope for a single-client rebuild and should not be folded in. | Defer: file a follow-up to evaluate extracting the shared leniency/base-resource/error primitives into a common internal package once both clients are on the throwing model. | architect-r1-f7 | Low | Open | Performance | R3 (l.76); pagination behavior | `Defer:` `paginate` eagerly walks every page and buffers the **full** result set in memory (e.g. 1500+ resolved alerts today, unbounded as accounts grow), with no streaming/async-iterator escape hatch on the frozen `1.0.0` surface. This matches existing 0.1.x behavior and is within the design's stated intent, so it is not something to fold in now. | Defer: file a follow-up to consider an async-iterator / page-cursor variant for large collections. No change to this design; recorded so the eager-buffer choice is a known, revisitable tradeoff rather than an implicit one. |
