| project-lead-r1-f1 | Blocker | Open | Requirements | `spec/openapi.json` (defect), fix belongs in `scripts/patch-spec.mjs`; symptom in `src/generated/schemas/**` | Datto's spec has a systemic documentation bug: for **51 of 57 operations** (35 of them `GET` reads ‚Äî `getByUid`, `getUserAccount`, `getSite`, `getSites`, `getUserAccountDevices`, `getAlert`, every job/audit/filter read, etc.) the spec has **no `200`/`204` response entry at all** ‚Äî the real success schema is instead misattached to an error code (`401`/`403`/`404`, sometimes `400`), verified directly (e.g. `GET /v2/device/{deviceUid}` declares only `401/403/404/409/500`, and its `401`/`403`/`404` entries all `$ref` `#/components/schemas/Device` ‚Äî the obvious real 200 body, just filed under the wrong code). Because Orval's zod target only emits a runtime response validator when an operation's own `200`/`204` entry has a schema, **the committed `src/generated/schemas/**` has zero response validators for these 51 operations** (confirmed by direct inspection: across all 9 tag files only `getActivitiesResponse`, `getStatusResponse`, `getByMacAddressResponse`, and `getDeviceAuditByMacAddressResponse` exist ‚Äî every other exported name is a `*Params`/`*Body` request-side schema). This is invisible to this phase's exit gate (typecheck/lint/test/reproducibility all pass ‚Äî a missing schema isn't a compile error), but it directly breaks the architecture the plan itself assumes downstream: Phase 6 Step 3 says the schema-override module "imports generated zod schemas" and "re-composes" the **generated** `Device` response schema with `udfSchema`, and Phase 7's own example (`this.httpGet(..., deviceResponseSchema, ...)`) presumes a generated `deviceResponseSchema` exists ‚Äî for `GET /device/{uid}` (and 34 other reads spanning account/site/alert/job/audit/filter) it does not. This is squarely a "known spec defect generation cannot infer" (the class Phase 2's patch step exists to fix, same as `TIMESTAMP_FIELDS`/`REQUEST_RESPONSE_SPLITS`), not a Phase 6 concern, and it is far more consequential than any of the three corrections `patch-spec.mjs` already makes ‚Äî R4 ("response schemas generated by Orval") is not met for ~89% of the surface. | In `scripts/patch-spec.mjs`, add a deterministic correction that, for every operation whose `responses` lacks a `200`/`204` entry but has an error-code entry (`400`/`401`/`403`/`404`) carrying a `content` schema, synthesizes the missing success response from that schema (the misattributed entries are consistent per-operation, so picking any one of them ‚Äî or the most complete ‚Äî is safe). Fail loud only where truly nothing is inferable (verify each such case is a legitimate void write, e.g. `setUdfFields`, before treating an empty response as intentional). Re-run `npm run generate` and confirm the previously-empty tag files (`-v2-site`, `-v2-account`, `-v2-job`, `-v2-audit`, `-v2-filter`, most of `-v2-device`/`-v2-alert`) now emit response schemas for their read operations, and add a `tests/generated/patch-spec.test.ts` case covering the synthesis.
