| engineer-r1-f1 | Medium | Open | Testing / Error Handling | `tests/generated/reproducibility.test.ts` l.23–30 | The R15 reproducibility guard runs `npm run generate` then `git diff --exit-code -- src/generated`, but `git diff` only compares **tracked** files against the index — it ignores untracked files and does not run against a clean-slate tree. So the two most likely reproducibility regressions slip through silently: (a) a spec change that makes Orval emit a *new* type file leaves an untracked file that `git diff --exit-code` does not report (test passes falsely); (b) a spec change that *removes* a type leaves an orphaned committed file that a from-scratch generate would not produce, again undetected. The phase notes state the real check was `rm -rf src/generated && npm run generate`, but the automated regression test omits the `rm -rf`, so it verifies strictly less than the property it claims to guard (byte-for-byte reproduction from scratch). | Before generating, `rm -rf src/generated` (the test is skip-guarded on a committed spec and regenerates the tree anyway, so the destructive step is safe and restores state); and assert on `git status --porcelain -- src/generated` being empty (which catches untracked *and* deleted files) rather than `git diff --exit-code`, which ignores both. Alternatively generate into a temp dir and compare trees. |
| engineer-r1-f2 | Low | Open | Dead Code | `scripts/widen-response-enums.mjs` `widenGeneratedTypes` l.343, 350 | `const fileNamesByPrimaryName = new Map()` is built (`fileNamesByPrimaryName.set(name, fileName)`) but never read anywhere in the function or module — it is dead state that suggests an intended-but-unused mapping and misleads a maintainer into thinking file identity is tracked for exclusion. | Delete the `fileNamesByPrimaryName` declaration and its `.set(...)` call. |
| engineer-r1-f3 | Low | Open | DRY / Complexity | `scripts/widen-response-enums.mjs` `widenGeneratedTypes` l.346–355 and l.360–367 | Each generated file is parsed by `parseGeneratedFile` **twice** — once in the first loop to collect `importNames`/`primaryNames`, and again in the second loop solely to recompute `primaryNames` for the exclusion check. The second parse is redundant work and duplicates the intent of the first. | In the first loop, cache each file's `primaryNames` (e.g. a `Map<fileName, Set>` alongside the content map, or store `{content, primaryNames}`), and reuse it in the second loop instead of re-calling `parseGeneratedFile`. |
| engineer-r1-f4 | Low | Open | Error Handling / Robustness | `scripts/widen-response-enums.mjs` `IMPORT_RE` (l.279), `ENUM_ALIAS_RE` (l.282), `main()` (l.386–416) | The whole request/response discrimination is load-bearing for safety (wrongly widening a request enum silently loosens the compile-time request contract — the R6 hazard), yet it rests entirely on regexes hard-coded to Orval's exact current output style (single-quoted imports, exact enum-alias one-liner). If a future Orval/formatter change alters that style (e.g. double quotes), `IMPORT_RE`/`ENUM_ALIAS_RE` silently stop matching → transitive exclusion under-expands → request enums get widened, with **no error** and only a plausible-looking `changedCount` in the log. There is no assertion that the discrimination matched anything it was expected to. | Add a cheap invariant in `main()` after building the exclusion set: e.g. when `requestOnlyComponentNames.size > 0`, assert `rootExcluded`/`excluded` is non-empty (a known request root must resolve to at least itself), or assert at least one enum alias was actually widened — so a silent no-match fails loud instead of degrading safety. |
| engineer-r1-f5 | Low | Open | DRY | `scripts/patch-spec.mjs` `walkAllSchemaNodes` l.183–199; `scripts/widen-response-enums.mjs` `collectComponentRefs` l.119–154 and `schemaHasEnum` l.163–186 | The JSON-Schema traversal skeleton — recurse into `properties.*`, `items`, object-typed `additionalProperties`, and each of `allOf`/`oneOf`/`anyOf` — is hand-reimplemented three times across the two scripts (including the literal `["allOf","oneOf","anyOf"]` keyword list appearing three times). The three differ only in their per-node action ($ref-follow vs identity-visit vs enum-detect), so the recursion shape is genuinely duplicated and a keyword added in one place is easy to forget in the others. | Extract one small `forEachSubschema(node, visit)` (or equivalent generic walker) into a shared helper the scripts import, and express each of the three as its own `visit` callback over that walker; keep the `$ref`-following vs not distinction as a parameter. If a shared module between the two scripts is judged to add more coupling than it removes, at minimum hoist the `["allOf","oneOf","anyOf"]` keyword list to a single named constant referenced by all three. |
