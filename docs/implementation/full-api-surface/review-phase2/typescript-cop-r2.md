| typescript-cop-r2-f1 | High | Open | TypeHole | `scripts/lib/schema-walk.mjs` (`SchemaNode`, `OpenApiOperation`, `OpenApiSpecFragment` typedefs, l.13-41), consumed by `scripts/patch-spec.mjs`, `scripts/widen-response-enums.mjs`, and every fixture in `tests/generated/patch-spec.test.ts` / `tests/generated/widen-enums.test.ts` | The shared typedefs added to close `typescript-cop-r1-f1` each carry a blanket `[key: string]: unknown` index signature. Verified directly (not assumed): a fresh `OpenApiOperation` literal with a typo'd key — `{ respnoses: { "200": {...} } }` instead of `responses` — type-checks with **zero** diagnostics under `npx tsc --noEmit -p tsconfig.test.json`, and the same holds for a spec-fragment literal with `requestBdoy` instead of `requestBody` passed directly into `verifyNoSharedEnumBearingSchemas`. This is because an index signature makes *every* string key a valid member of the type (value `unknown`), which unconditionally suppresses TypeScript's excess-property checking for that type — not just for genuinely-unmodeled real-spec fields (`title`, `description`, `default`, …, which is the index signature's legitimate purpose) but for near-miss typos of the *modeled* keys (`responses`, `requestBody`, `properties`, `$ref`, `allOf`, …) too. This reopens the exact hazard `typescript-cop-r1-f1` was raised to close — "test fixtures … get zero structural checking against what the function actually expects" — through a new mechanism: the fix now gives every parameter a concrete, non-optional type, but that type's own permissiveness means a mistyped fixture (e.g. `responses` → `respnoses` in one of `patch-spec.test.ts`'s `withOperation`-built operations, or `requestBody` → `requestBdoy` in one of `widen-enums.test.ts`'s inline `specWith(...)`-style literals) still silently type-checks as valid input, silently no-ops the function under test, and is caught only if the specific test happens to assert a *positive* outcome (`.toThrow()`) that the no-op would then fail to produce — a `.not.toThrow()` case (e.g. `widen-enums.test.ts`'s "does not throw when a shared component is enum-free") would keep passing even though the mistyped fixture never actually exercised the shared-schema path it claims to. | Since `SchemaNode`/`OpenApiOperation`/`OpenApiSpecFragment` need their index signatures for their one real caller — `main()`'s `JSON.parse(readFileSync(SPEC_PATH, …))` result, which is never a literal and gets `any` from `JSON.parse` regardless — keep the permissive typedefs for that use, but stop using them as the type for hand-written test-fixture object literals. Define closed (no `[key: string]: unknown`) fixture-only variants — e.g. `type StrictSchemaNode = Omit<SchemaNode, never> & { [K in keyof SchemaNode]?: SchemaNode[K] }` won't help (still structurally compatible via the same index signature) — the workable fix is a genuinely separate, closed interface per fixture shape (mirroring what `patch-spec.test.ts`'s own local `SpecFragment` almost does, except it also carries `[key: string]: unknown` at l.25 and should drop it), and use those closed types — not the shared permissive ones — when declaring `buildValidSpecFragment()`'s return type, `withOperation`'s `operation` parameter, and every ad-hoc `spec`/operation literal in `widen-enums.test.ts`. This restores excess-property checking exactly where it matters (hand-typed test fixtures) without weakening the real spec-fragment types the production code needs to stay permissive. |
| typescript-cop-r2-f2 | Low | Open | PublicTypes | `scripts/patch-spec.mjs` (`EXPECTED_ORPHANED_COMPONENTS` l.133-162, `VOID_WRITE_OPERATIONS` l.175-186, `TIMESTAMP_FIELDS` l.68-72, `REQUEST_RESPONSE_SPLITS` l.98-121), `scripts/widen-response-enums.mjs` (`REQUEST_ROOT_SUFFIXES` l.102-112), `scripts/lib/schema-walk.mjs` (`SUBSCHEMA_KEYWORDS` l.44) | These exported `const` arrays/objects are the anchor/allow-lists that gate the module's fail-loud safety checks (an item missing from `EXPECTED_ORPHANED_COMPONENTS` is what makes an undocumented orphan throw instead of silently deleting a live component; membership in `VOID_WRITE_OPERATIONS` is what lets an operation pass as an intentional void write instead of failing loud). None is declared `as const` or typed `readonly`, so every importer — including the two test files that already import several of them directly (`TIMESTAMP_FIELDS`, `REQUEST_RESPONSE_SPLITS`, `EXPECTED_ORPHANED_COMPONENTS`) — gets a mutable `string[]`/plain object reference to the module's own singleton list; a future test (or any other consumer) that does `EXPECTED_ORPHANED_COMPONENTS.push(...)` or `VOID_WRITE_OPERATIONS.length = 0` for local convenience would silently corrupt the real anchor list for every other caller in the same process, including the CLI's own `main()` if it runs in the same run (e.g. under `test:repro`, which shells out to a fresh `node` process, so the immediate risk is contained today, but nothing in the type declares the intent or prevents it in-process). | Add `as const` to each of `TIMESTAMP_FIELDS`, `REQUEST_RESPONSE_SPLITS`, `EXPECTED_ORPHANED_COMPONENTS`, `VOID_WRITE_OPERATIONS`, `REQUEST_ROOT_SUFFIXES`, and `SUBSCHEMA_KEYWORDS`, so their exported types are readonly tuples/arrays and any accidental mutation at a call site becomes a compile error instead of a silent shared-state corruption. |
