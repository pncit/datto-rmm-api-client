## typescript-cop — round 3

Scope: the working tree's unstaged changes since the round-2 checkpoint, i.e. the reviser's
round-3 disposition of `architect-r2-f1/f2`, `engineer-r2-f1/f2/f3`, and `typescript-cop-r2-f1/f2` —
`scripts/lib/schema-walk.mjs`, `scripts/patch-spec.mjs`, `scripts/widen-response-enums.mjs`, the new
`tests/generated/strict-fixture-types.ts`, and the corresponding `tests/generated/patch-spec.test.ts`
/ `widen-enums.test.ts` edits. Verified, not assumed: re-ran `npx tsc --noEmit -p tsconfig.test.json`
(clean) and several throwaway probe files/edits (reverted after each check) exercising the actual
compiler behavior the disposition claims, rather than trusting its prose — see each finding below for
the exact probe and result.

`typescript-cop-r2-f2` is ratified: probing confirmed `EXPECTED_ORPHANED_COMPONENTS.push(...)`,
`VOID_WRITE_OPERATIONS.push(...)`, and `REQUEST_ROOT_SUFFIXES.push(...)` are now all compile errors
(`TS2339: Property 'push' does not exist on type 'readonly string[]'`).

`typescript-cop-r2-f1` is only partially closed — the `SchemaNode`/`OpenApiOperation` leaf-level fix
is verified working, but a probe found the fix stops one level too shallow (the spec-fragment
container itself is still unguarded); kept `Open` below with the counter-evidence and a verified
working extension of the same technique.

Probing further, independent of any prior finding, surfaced a new and substantial gap: the whole
"type the `.mjs` scripts via JSDoc" strategy this phase's `typescript-cop-r1-f1`/`r2-f1` fixes rest
on only protects **external callers** of the four-plus *exported* functions (what a test file sees
when it imports them) — it provides **zero** protection for the dozens of *internal*, non-exported
helper functions inside `patch-spec.mjs`/`widen-response-enums.mjs`/`dedupe-generated-index.mjs`,
because `checkJs` was never enabled anywhere in the toolchain. See `typescript-cop-r3-f1`.
