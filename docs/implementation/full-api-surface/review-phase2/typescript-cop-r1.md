## typescript-cop — round 1

Scope: `git diff 9c8dd39` (the Phase 1 commit) → current tree, i.e. Phase 2's own additions only —
`scripts/patch-spec.mjs`, `scripts/dedupe-generated-index.mjs`, `scripts/widen-response-enums.mjs`,
`tests/generated/*.test.ts` (4 new suites), and the `tsconfig.test.json`/`eslint.config.js` deltas
that support them. `src/generated/**` and `spec/**` are Orval/fetch output, not hand-written, and are
not yet wired into the package's public export surface (`src/index.ts` is unchanged) — reviewed only
enough to confirm that (no new exports, no boundary consuming them yet), per the skill's "generated
code is never hand-fixed" framing already established for this phase. No prior `typescript-cop` turn
exists for this phase, so this is a fresh, exhaustive first pass.

The scripts themselves are careful, defensively-written JS: fail-loud anchor checks for the two named
spec corrections and the `ProxySettings` split, cycle-safe `$ref` walks, a pure-core/thin-CLI-wrapper
split in all three, and a genuinely necessary (not gold-plated) two-mechanism request/response
discrimination in the widen codemod, all independently verified by reading the code against the
module docs' own claims. One systemic gap: three of the four exported script functions the new tests
exercise carry no type information across the JS→TS boundary `tsconfig.test.json`'s `allowJs` addition
was specifically built to check (verified directly with `tsc`, not assumed — see f1).

## Findings

| ID | Severity | Status | Category | Location | Finding | Recommendation |
|----|----------|--------|----------|----------|---------|----------------|
| typescript-cop-r1-f1 | High | Open | TypeHole | `scripts/patch-spec.mjs` (`patchSpec`), `scripts/widen-response-enums.mjs` (`computeRequestOnlyComponentNames`, `verifyNoSharedEnumBearingSchemas`, `widenGeneratedTypes`) | These four exported functions — the core Phase 2 pipeline logic the new `tests/generated/*.test.ts` suites exercise — carry no JSDoc `@param`/`@returns` types, so under `tsconfig.test.json`'s new `allowJs: true` + `scripts/**/*.mjs` include (added this phase specifically so "tests can import the … generate scripts" and so `tsc` can "read the scripts' actual JS structure for inferred types", per the phase notes' own deviation #4), TypeScript infers every parameter as implicit `any` and, for three of the four, the return type is `any`/`Set<any>`/`Map<any, any>` too. Verified directly (not assumed): `Parameters<typeof patchSpec>` is `[spec?: any]`, `ReturnType<typeof patchSpec>` is `any`, `Parameters<typeof computeRequestOnlyComponentNames>` is `[spec?: any]` with `ReturnType` `Set<any>`, `Parameters<typeof verifyNoSharedEnumBearingSchemas>` is `[spec?: any]`, and `Parameters<typeof widenGeneratedTypes>` is `[fileContentsByName?: any, requestOnlyComponentNames?: any]` with `ReturnType` `Map<any, any>`. Consequences that are not hypothetical: every parameter is inferred *optional* (`spec?: any`), so `npm run typecheck:test` would not catch a call to `patchSpec()`/`verifyNoSharedEnumBearingSchemas()` with zero arguments even though both dereference `spec` unconditionally; test fixtures built in `patch-spec.test.ts`/`widen-enums.test.ts` (e.g. the ad-hoc `spec` object literals in the `verifyNoSharedEnumBearingSchemas` describe block) get zero structural checking against what the function actually expects; and `widened.get(fileName)` in `widen-enums.test.ts` returns `any`, so a typo'd file-name key silently type-checks instead of surfacing as `string | undefined`. This is a real regression introduced by this phase, not a pre-existing gap: `scripts/dedupe-generated-index.mjs`'s own `dedupeExportLines` (in the same phase, same interop mechanism) *does* carry `@param {string} content` / `@returns {{content: string, duplicatesRemoved: number}}` JSDoc and is confirmed correctly typed (`Parameters<typeof dedupeExportLines>` is `[content: string]`, non-optional) — proving the fix is already the established pattern in this same diff, just not applied to the other three, more consequential, functions (one of which, `verifyNoSharedEnumBearingSchemas`, is the regression guard for a real production hazard already found once this phase). | Add JSDoc `@param`/`@returns` annotations to `patchSpec`, `computeRequestOnlyComponentNames`, `verifyNoSharedEnumBearingSchemas`, and `widenGeneratedTypes`, matching the precedent already set by `dedupeExportLines` — at minimum typing `spec` as the OpenAPI-fragment shape the test file's own `SchemaNode`/`SpecFragment` interfaces already describe (a shared `@typedef` importable by both the `.mjs` scripts and the `.ts` tests would let both sides use the same type instead of duplicating it), and `fileContentsByName`/`requestOnlyComponentNames`/the `Map`/`Set` return types explicitly (`Map<string, string>`, `Set<string>`). This makes the parameters non-optional and removes the `any` leak into every call site in `tests/generated/*.test.ts`. |
