## reviser — round 12

Internal self-review (Phasing, Specificity, Exit gates, Edge cases, Hidden complexity, Dependencies,

| ID | Disposition | Rationale |
|----|-------------|-----------|
| architect-r7-f1 | Accept | The Phase 2 widen-guard was shallow — it inspected only the top-level `requestBody` `$ref`, but the codemod widens **nested** component-schema types too, so a `*Body` reaching a shared component schema *transitively* (via a nested property `$ref`) would silently over-widen that shared enum on the request-side TS type. Rewrote Phase 2 Step 4 (l.159) so the guard **resolves `$ref`s recursively** (through `properties`/`items`/`allOf`/`oneOf`/`anyOf`/`additionalProperties`, with a visited-set for cycles), collects the **full transitive** `#/components/schemas/*` set reachable from every request body and from every response, and throws if they intersect — making the guard verify the assumption at every depth. Extended `widen-enums.test.ts` (l.201) with a second case proving a **transitive/nested** shared ref also throws. |
| engineer-r7-f1 | Accept | The Phase 9 recursive `WIDENED_FIELDS` completeness guard needs to enumerate every enum node at arbitrary depth, which requires `_zod.def` introspection — a second parallel site outside `schema-leniency.ts` would defeat the single-isolation-boundary mitigation. Made `enumFieldPaths(schema)` an **exported helper of `src/validation/schema-leniency.ts`** built on the existing `addCatchallRecursive`/`detectUnknownProperties` walk (Phase 4 Step 1, l.306), and stated in Phase 9 Step 3 (l.609) and the example (l.629) that the guard imports it from there. All `_zod.def` access stays in one file and the guard cannot desync from the walker that widens enums at runtime. |
| engineer-r7-f2 | Accept | After r11 the `*_ENUM_FIELDS` constant lists **containing top-level properties** (`antivirus`, `patchManagement` are sub-objects, not enums), so the name contradicted its contents and could mislead an implementor into listing nested enum names (won't compile) or just `['deviceClass']` (revives the nested-enum-typed-closed bug). Renamed `DEVICE_ENUM_FIELDS`/`ALERT_ENUM_FIELDS`/`ENUM_FIELDS` → `DEVICE_WIDENED_FIELDS`/`ALERT_WIDENED_FIELDS`/`WIDENED_FIELDS` throughout (Phase 6 Step 3, Phase 9 Step 3 + example), and added a one-line note at l.436 stating each entry is the **containing top-level property** whose subtree holds an open enum (including nested), not the enum field itself — so the name matches the recursive guard's semantics. |
