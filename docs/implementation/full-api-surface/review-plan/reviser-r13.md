| engineer-r8-f1 | Accept | The Phase 9 completeness-guard example iterated `Object.entries(WIDENED_FIELDS)`, binding `entity` to a name string, then passed it to `enumFieldPaths` — which introspects `_zod.def` and requires a `z.ZodType`, not a string, so the guard was unimplementable without either a layering inversion (`schema-leniency.ts` reverse-depending on the Phase 6 override registry) or a nonexistent name→schema map. | engineer-r8-f2 | Accept | The transitive widen-guard threw on any request/response component-name-set intersection with no check that the shared component carries an enum. Since widening is a no-op on enum-free schemas and the recursive walk now dives into deep, widely-shared sub-objects (address/pagination/metadata), a benign share would fail `npm run generate` for nothing. Restricted the throw (Phase 2 Step 4, l.159) to a shared component **whose own or a nested schema declares an `enum`** — precisely when over-widening is possible — and left enum-free shares silent. Updated `widen-enums.test.ts` (l.202): the two throw cases now use enum-bearing shared components, and added case (iii) proving an enum-free share does **not** throw. |
| engineer-r8-f3 | Accept | The fail-loud throw existed to "force an explicit fix" but had no message contract, unlike the sibling `patch-spec` fail-loud that names its missing anchors. Added a message-contract bullet to Phase 2 Step 4: the guard's `throw new Error(...)` **names the offending intersecting `#/components/schemas/*` component(s)** and the request-body/response operation(s) that reach each, so the maintainer gets a self-locating pointer to the schema to split. Added a test assertion (l.202) that the thrown message names the offending component(s), mirroring the patch-spec drift-message test. |
