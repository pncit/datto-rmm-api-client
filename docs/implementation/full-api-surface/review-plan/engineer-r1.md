## engineer — round 1

Plan Review Mode, round 1 (fresh — no prior `engineer` turn files in this review dir). Reviewed
`plan.md` against `design.md` across the five plan-review axes: (a) DRY & reuse, (b) abstraction &
complexity, (c) error handling & logging, (d) naming & intent, (e) configuration & magic values.

The plan is unusually thorough and mostly well-decomposed (clean coexistence rule, faithful ports,
byte-reproducibility gate, named default constants). Findings below are the real gaps that would
cause rework or a broken exit gate if implemented as written. Architecture-level and
requirements-coverage concerns (e.g. R17 real-vs-synthesized fixtures) are deferred to
`/architect` and `/project-lead` respectively and not raised here.

## Findings

| ID | Severity | Status | Category | Location | Finding | Recommendation / update |

| engineer-r1-f1 | High | Open | ErrorHandling | Phase 5 Step 3 & Phase 6 Step 1–2 ("documented `rateDescriptor` property") | The plan threads a custom `rateDescriptor` property on the axios request config (attached by `http*` primitives and by `paginate`, read by the request interceptor). Axios's `AxiosRequestConfig`/`InternalAxiosRequestConfig` do not declare this property, so both the attach sites (`this.axios.get(url, { rateDescriptor })`) and the interceptor read (`config.rateDescriptor`) will fail `tsc --noEmit` — breaking every phase-boundary typecheck exit gate from Phase 5 on. | Specify a `declare module 'axios'` (or a typed wrapper config) augmentation adding `rateDescriptor?: RateDescriptor` to `InternalAxiosRequestConfig`, name the file it lives in (e.g. `src/http/axios-augment.d.ts` or top of `http-client.ts`), and reference it in Phase 5 Step 3. |
| engineer-r1-f2 | Medium | Open | Complexity | Phase 1 Step 3, Phase 5 Step 3 (`src/http/axios-mutator.ts` + generated `endpoints/api.ts`) vs Phase 6 Step 1 (BaseResource "shared axios instance") | The relationship between the generated axios endpoints (`src/generated/endpoints/api.ts`) + `customInstance` mutator and BaseResource's own shared axios instance is unspecified. Resources call `this.httpGet(path, schema)` with hand-written paths (Phase 7 example), never the generated endpoint functions — so the endpoints target and `customInstance` appear to be dead, or there are two axios instances with two interceptor chains (only one of which has the auth/rate-limit/retry wiring). This is a real correctness/DRY ambiguity: a request routed through the wrong instance bypasses the limiter and auth. | State explicitly that the generated endpoints layer is an unused Orval artifact (kept only to emit types) and that `customInstance`/`http-client.ts` resolve to the **one** interceptor-bearing instance BaseResource uses — or, if endpoints are intended to be called, show it. Ensure a single authoritative axios instance. |
| engineer-r1-f3 | Medium | Open | ErrorHandling | Phase 3 Step 1 vs Phase 5 Step 4 example vs Phase 6 Step 2 example | `DattoValidationError`'s constructor shape is inconsistent across phases: Phase 3 defines it with `zodError`, `stage`, and an "optional wire payload"; Phase 5 constructs `new DattoValidationError(err, 'request')` (2-arg); Phase 6 constructs `new DattoValidationError(cursor.error, 'response', { context })` (3rd arg is an options bag carrying `context`, not a wire payload). The implementor cannot tell whether arg 3 is the wire payload, an options object, or both, and where `context` lives. | Pin one constructor signature in Phase 3 (e.g. `(zodError, stage, opts?: { payload?: unknown; context?: string })`) and make the Phase 5/6 examples conform. |
| engineer-r1-f4 | Medium | Open | ErrorHandling | Phase 3 Step 1 vs Phase 5 Step 3 example | `DattoApiError` is described in Phase 3 only by its fields (`statusCode`, `response`, `requestId`, `retryAfterMs?`, `code?`) plus `static fromAxiosError`, but the Phase 5 example calls `new DattoApiError('IP block', { statusCode, code, response, cause })` — a message + options-bag constructor that is never defined, and uses a `cause` field not listed among the type's fields. | Define the public constructor signature in Phase 3 and add `cause` (or `Error.cause`) to the documented field list so the direct-construction path (403) and `fromAxiosError` agree. |
| engineer-r1-f5 | Medium | Open | ErrorHandling | Phase 2 Step 2 + sketch (`scripts/patch-spec.mjs`, `if (props?.[f])`) | The patch step silently no-ops when an expected timestamp field is absent (`if (props?.[f])`). A future spec refresh that renames/relocates a timestamp field would silently reship it as `string` (re-introducing the exact defect R8 exists to fix), relying only on downstream fixture tests to notice. The cheaper, more direct guard is a fail-loud patch step. | Have `patch-spec.mjs` throw (or emit a non-zero exit) when a field in `TIMESTAMP_FIELDS` (or the `Alert.alertContext` anchor) is missing from the fetched spec, so a `generate` after a drifting refresh fails at the patch step. Cover with a unit case in `patch-spec.test.ts`. |
| engineer-r1-f6 | Medium | Open | Complexity | Phase 2 Step 4 (`widen-response-enums.mjs` suffix heuristic) | The response/request discrimination rests entirely on the unverified load-bearing assumption "Datto's write bodies are distinct small `*Body` types (no shared component-schema is used as a request body)". If any request body aliases a component schema (e.g. a `Device`-shaped body), the codemod widens that component's enums and the request-body **type** silently loosens. The Repo Context confirms enum emission shape but does not confirm this assumption. | Add a Phase 2 verification step (or an assertion in the codemod/test) that no request body in the patched spec `$ref`s a component schema also used as a response — or scope widening by actual response-schema membership rather than by name suffix. |
| engineer-r1-f7 | Medium | Open | DeadCode | Phase 8 Step (coverage-map), `tests/unit/client/coverage-map.test.ts` / `src/client/operation-map.ts` | The coverage-map test asserts the maintained `client.<ns>.<method> → {method,path}` table covers every spec operation, but nothing asserts the **resources actually implement** the mapped paths. A resource method with a wrong or missing path passes R1's "real guard" as long as the hand-maintained map is correct — and the map duplicates path/verb strings already hard-coded in each resource body, so the two will drift. | Either derive the map from resource metadata (single source of truth) or add an assertion that each mapped operation is reachable via the constructed client at the declared path/verb (e.g. via nock intercept), so the guard verifies resources, not just the map. |
| engineer-r1-f8 | Low | Open | DeadCode | `plan.md` end of file (lines 628–629) | The plan file ends with stray `</content>` and `</invoke>` lines — leftover tool-call markup accidentally committed into the document. | Delete the two trailing artifact lines. |
| engineer-r1-f9 | Low | Open | Naming | Phase 8 Step 8 & Phase 9 Step 1 | Fixtures live in two homes after the rebuild: the retained real captures stay in `src/__tests__/fixtures/` (Phase 8 keeps them) while all new fixtures go to `tests/fixtures/` (Phase 9). | engineer-r1-f10 | Low | Open | MagicValues | Phase 1 Step 1 (`@types/node@^26`) | `@types/node@^26` is pinned while the supported runtime floor is Node ≥ 20 (Non-Goals, R16). Typing against Node 26 lets code compile against APIs not present on Node 20, defeating the floor. | Pin `@types/node` to the supported floor (e.g. `^20` or `^22`) rather than mirroring fuze-api's `^26`. |
| engineer-r1-f11 | Low | Open | MagicValues | Phase 3 Step 4 (`DEFAULT_RETRY`), Phase 5 Step 4 (`DEFAULT_TOKEN_REFRESH_PCT`), Phase 5 Step 1 (rate-limit consts) | Default constants are scattered across three modules with three ad-hoc rationales: retry defaults in `datto-client-config.ts`, token-refresh default in `auth-manager.ts`, rate-limit defaults in `rate-limits.ts`. The retry default in particular is consumed by the http-client, not the config. | Co-locate each default with its consumer consistently, or centralize the cross-cutting numeric defaults in one `src/client/defaults.ts` and import from there, so the "one source" rule the plan repeatedly invokes is applied uniformly. |
| engineer-r1-f12 | Low | Open | ErrorHandling | Phase 5 Step 3 & example (429 handling) | `Retry-After` is parsed only as numeric seconds (`Number(header)`); the RFC-permitted HTTP-date form yields `NaN` and silently falls back to computed backoff, ignoring the server's explicit instruction (R12 says "honors server 429 Retry-After"). | Handle the HTTP-date form of `Retry-After` (or explicitly document that only delta-seconds is supported and why Datto only sends that), and add a nock test for the non-numeric case. |
| engineer-r1-f13 | Low | Open | Logging | Phase 3 Tests (`mask.test.ts`) | The masking test covers string and object udf values but not a **numeric** udf value, though Phase 3 Step 3 and R20/design explicitly require masking "regardless of wire type (string, number, or a nested object/array)". A regression that passes numbers through would go uncaught. | Add a numeric-udf case (e.g. `udf3: 12345 → '[redacted - 5 characters]'`) to the mask test. |
| engineer-r1-f14 | Low | Open | Naming | Phase 5 Step 3 (`isRateLimitBlock(response)`) | The `isRateLimitBlock(response)` predicate is central to the 403 `ip-block`/`forbidden` split but is never assigned a home module — described only as "a documented predicate". Left unplaced it risks being inlined ad hoc at the 403 site. | Name the file it lives in (e.g. exported from `http-client.ts` or `src/errors/`) so it is one documented, testable source consumed by the error path. |

It also overstates its own importance: since `spec/openapi.json` is committed once and frozen, reproducibility derives from the frozen file, not from re-sorting. | Reword to state the single requirement (a stable, deterministic serialization committed once) and drop the contradictory "sorted … as fetched". |
| engineer-r1-f16 | Low | Open | Naming | Phase 6 Step 2 prose vs example signature | The `paginate` parameter order in prose ("given a start path, params, the page's named-array key + item schema") disagrees with the example signature `paginate(startPath, arrayKey, itemSchema, params?, context?)`. An implementor following the prose would order the args wrong. | Align the prose to the example signature (or vice versa) so the parameter order is unambiguous. |
