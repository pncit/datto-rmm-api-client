## typescript-cop — round 1

Scope: the Phase 4 diff (`src/validation/schema-leniency.ts`, `src/validation/diagnostics.ts`, and
their test files) against `main`, read against the current working tree (post the reviser's round-1
fixes to `implementation-auditor`'s findings, which are out of this review's scope — I evaluate the
code as it stands, not that history). Cross-checked `def`-property names actually read off Zod's
untyped internal `_zod.def` (`getDef`) against `node_modules/zod/v4/core/schemas.d.ts` (`keyType`,
`entries`, `element`, `innerType`, `options`, `in`/`out`, `defaultValue` all verified present and
correctly named today), and cross-checked which `def.type` values the generated schemas actually
produce (`grep -ohE "zod\.[a-zA-Z]+\(" src/generated/schemas/**/*.zod.ts`: only `array`, `boolean`,
`enum`, `number`, `object`, `record`, `strictObject`, `string` appear).

The dominant concern is the mismatch between `parseLenient`'s static return type and what it actually
hands back on success once nullability leniency has applied — everything else is secondary hardening.

| typescript-cop-r1-f1 | High | Open | TypeHole | `src/validation/schema-leniency.ts:597-629` (`parseLenient`), `:77-79` (`toLenientField`) | `parseLenient<T>(schema: z.ZodType<T>, …)` returns `z.ZodSafeParseResult<T>`, and on success does `return { success: true, data: cleaned as T }`. `T` is inferred from the *original* schema — and every generated response schema under `src/generated/schemas/**` marks its fields `.optional()` but never `.nullable()` (confirmed: `grep -c "\.nullable()" src/generated/schemas/-v2-device/-v2-device.zod.ts` → 0), so `T`'s corresponding TS type (`src/generated/types/device.ts`: `hostname?: string`, etc.) is `string \| undefined`, never `null`. But `toLenientField` (`fieldSchema.nullable().optional()`) is applied to *every* named object field at *every* depth specifically so a response `null` value tolerates and parses successfully — meaning a real, common case (`{ hostname: null }`) type-checks at the call site as `Device` (`hostname: string \| undefined`) while the runtime value is actually `null`. A consumer who writes the idiomatic `if (device.hostname !== undefined) { device.hostname.toUpperCase(); }` — a check the declared type invites — throws `TypeError: Cannot read properties of null` at runtime on data the type claimed was excluded. This is exactly the "unvalidated external input treated as strongly typed" pattern the severity floor names; unlike the enum-widening leniency (R5's codegen step honestly widens the *generated type* to `EnumUnion \| (string & {})` so the compile-time contract matches the runtime relaxation), the nullability leniency added in this phase has no analogous type-level signal at all — the design's risk register accepts the *runtime* risk ("masks a genuinely missing required field… mitigation: … per-item drops are logged … not silent"), but that mitigation is the not-yet-built Phase 6 drop path; today's silent, successful `null` substitution (verified by the phase's own tests: "all with zero diagnostics — this is silent tolerance") has no logging and no type signal, so nothing observes it. | Give `parseLenient`'s success type an honest shape instead of the bare cast to `T` — e.g. export a recursive `Lenient<T>` mapped type (`{ [K in keyof T]: T[K] extends object \| undefined ? Lenient<NonNullable<T[K]>> \| null : T[K] \| null }`, mapping through arrays element-wise) and change the signature to `z.ZodSafeParseResult<Lenient<T>>`, so every Phase 6 caller building on this primitive is statically forced to handle `null` on any field rather than inheriting a type that quietly lies. If the team intends to keep `T` as-is for ergonomics, that's a decision this phase's own code doesn't currently reflect anywhere (not even as a documented caveat on the return type itself) — escalate for an explicit ruling rather than leave the cast unexamined. |
| typescript-cop-r1-f2 | Medium | Open | Exhaustiveness | `src/validation/schema-leniency.ts:203-207` (`addCatchallRecursive` default), `:478-480` (`detectUnknownProperties` default) | Both functions switch on `def.type as string` — a string with no literal-union type behind it, since `getDef` returns `any` — and both fall back to returning the schema/data completely unmodified for any `def.type` not in their explicit list (currently: object/strictObject, array, union, optional, nullable, record, pipe, default, enum, plus a handful of terminals). Nothing in the generated schemas exercises this today, but it is exactly the "default case that hides a new variant" pattern: a future `src/schema-overrides.ts` (Phase 6) or a spec refresh that introduces e.g. `z.intersection`/`z.tuple`/`z.lazy` would silently opt that entire subtree out of catchall-stripping, nullability leniency, *and* enum widening — with zero signal that it happened, since the fallback is silent by design ("Defensive: unknown schema type -- return unchanged to avoid crashes"). | At minimum, emit a single one-time `debug`/dev-only diagnostic (or throw in a way test coverage would catch, mirroring the invariant-guard already added for the union/discriminator case) when `default` is hit with an unrecognized `def.type`, so an unhandled schema-node type fails loudly during development instead of silently disabling leniency for that subtree in production. |
| typescript-cop-r1-f3 | Medium | Open | TypeHole | `src/validation/schema-leniency.ts:54-56` (`getDef`) | `getDef` is explicitly annotated to return `any` (not `unknown` or a minimal structural type), and ~20 call sites across `addCatchallRecursive`, `detectUnknownProperties`, and `enumFieldPaths`'s `walk` read `def.type`, `def.element`, `def.innerType`, `def.options`, `def.entries`, `def.valueType`, `def.keyType`, `def.in`/`def.out`, `def.defaultValue` off it with zero compiler checking of the property names themselves. Today's names are all verified correct against `zod/v4/core/schemas.d.ts`, but this file is the one place the design explicitly designated as the sole safe location to catch a zod-internal-shape drift via type-checking ("Isolate all zod-internal access to `schema-leniency.ts`… cover leniency with unit tests") — and `any` defeats exactly that, since a future typo or a zod-minor-bump rename (e.g. `innerType` → `inner`) compiles silently and only fails at runtime, inside whatever test happens to exercise that branch. | Change `getDef`'s return type from `any` to a minimal structural type, e.g. `{ readonly type: string } & Record<string, unknown>` — `def.type` gets checked directly, and every other property access still needs (and already has) an explicit `as` cast at each site, but a renamed/misspelled property now produces `undefined` visibly through a typed `unknown`/cast path instead of silently through `any`, and a genuine typo in a *new* property name added later gets caught by the existing explicit casts rather than compiling clean. |
| typescript-cop-r1-f4 | Medium | Open | TypeHole | `src/validation/diagnostics.ts:38-44` (`groupKey`) | `groupKey` builds the `Map` dedup key via naive space-joining: `` `${message} ${field} ${value ?? ""}` ``. `field` is not always static/schema-derived — for the `record` case in `detectUnknownProperties`, the diagnostic field path is `` `${path}.${key}` `` where `key` comes directly from `Object.keys(parsedRecord)`, i.e. an arbitrary externally-supplied record key from response data (e.g. `operationWriteStatus`'s `zod.record(zod.string(), …)` in `src/generated/schemas/-v2-system/-v2-system.zod.ts`, and the design's own planned `Udf` override in Phase 6).
