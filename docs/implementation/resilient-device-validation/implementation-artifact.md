# Resilient Device Validation — Implementation Process Artifact

_This cycle implemented, across two phases, the resilient per-device validation described in the approved design/plan: `getAccountDevices()` now returns every conforming device and reports each divergent one as a structured, logged warning instead of failing the whole account on a single odd device._

## Genesis

The work traces to issue #13. In the Datto RMM client's default `strict` validation mode, a **single** device that diverged from `DeviceSchema` failed the **entire** `getAccountDevices()` call: `getAllPages()` validated each page as one unit, so one non-conforming device threw a `ZodError` that aborted the walk and discarded all other devices. This was not hypothetical — a downstream daily sync had failed 100% for one account for over two weeks, even though authentication and every HTTP call returned `200`; the account simply contained one device whose shape was stricter than the schema allowed. The only workaround, `validationMode: 'warn'`, flowed the entire raw payload through unvalidated and routed its diagnostic to `console.warn`, bypassing any configured logger so the drift was invisible.

The design's stance: a divergent device is a **signal that the package's schema has drifted from upstream and a maintainer must reconcile it** — surfaced through the consumer's own logger at error level and returned as a structured warning, never silently dropped and never allowed to take down the account. `strict` still means strict (a divergent device is not returned), but its rejection is scoped to that device. The public `Device` type and `DeviceSchema` were to remain unchanged (R4); relaxing the schema was explicitly a non-goal.

The plan split delivery into two phases so the risky client rewiring rode on a fully unit-tested primitive:
- **Phase 1** — turn the single `validate()` seam into logger-aware primitives: a `validate()` that routes its `warn` diagnostic through a `LoggerLike`, and a new generic `validateItems()` that validates an array element-by-element and partitions by mode, plus the shared `toProblemError`/`extractIdentity` builder and error-shape constants. No client wiring.
- **Phase 2** — wire those primitives into `client.ts`: a two-pass `getAllPages` (structural envelope hard-fail, then per-device drift handling), an internal un-barrelled `DevicesEnvelopeSchema`, the `getDeviceByUid` fail-hard path, and release-note docs.

## Outcome

Both phases landed. Phase 1 was committed as `e8dc461` ("Logger-aware validation seam + per-item helper"); Phase 2 as `cc55a7d` ("Wire resilient validation into the client"). The final implementation lives in `src/validation.ts`, the new `src/internal/devicesEnvelope.ts`, and `src/client.ts`, with the living plan at `plan.md` and design at `design.md`.

What a reader should take away:
- `getAccountDevices()` in `strict` mode returns `{ ok: true }` with all conforming devices even when some diverge; rejected devices land in `Result.warnings[]` as a `ProblemError` naming the device (id/uid) and the failing Zod path, and are logged at error level through the configured logger.
- A structurally malformed page envelope (non-object body, `devices` not an array, unparseable `pageDetails`/`nextPageUrl`) still hard-fails in `strict`/`warn` as a protocol error (`type: "validation-error"`), but not in `off`, whose raw-passthrough contract is preserved.
- No public type changed; the internal envelope schema is deliberately off the `src/index.ts` barrel. The suite finished green (37 tests) with both R4 exit-gate guards and the README doc-landing grep passing.

## Process at a glance

Two review phases, each converging cleanly. Phase 1 ran the longer: six reviser rounds against implementation-auditor (3 rounds), gate-runner (2), engineer (4), architect (4), project-lead (4), and typescript-cop (4). Phase 2 ran four reviser rounds against implementation-auditor (3), engineer (3), architect (3), project-lead (3), and typescript-cop (3). Notably, **both architect and project-lead raised zero findings in all of Phase 2** — the requirements and architecture were assessed as fully met with no scope creep, a signal that Phase 1 had absorbed the hard structural decisions. Findings throughout were overwhelmingly Low/Medium; there was one High (a failing exit gate), root-caused to a pre-existing defect rather than the phase's own work.

## Key findings

**The only hard failure was a pre-existing one.** Phase 1's exit gate failed on the first run (High): three tests in `deviceSchema.test.ts`/`devicesMethod.test.ts` broke because the shared JSON fixtures never conformed to the unmodified `DeviceSchema` — every fixture set only `udf.udf1` (the schema requires `udf1`..`udf30`, each `string | null`) and used ISO-8601 date strings where the schema requires epoch-millisecond numbers. The reviser proved via a scratch worktree at the plan-approval commit that the failure predated any Phase 1 code, then corrected the four **fixture** files (not the protected schema) to match the existing contract, leaving all test assertions unchanged. The gate passed on re-run.

**Establishing single sources of truth for the shared error shape.** Both engineer and architect independently flagged that the first-failing-path computation (`issues[0]?.path?.join(".") || "(root)"`, including the `"(root)"` sentinel) was duplicated, with the plan directing a third copy into Phase 2 — resolved by extracting an exported `firstIssuePath` helper. The three ad-hoc log-message prefixes were consolidated into `VALIDATION_WARN_PREFIX`/`VALIDATION_ERROR_PREFIX` constants; architect later caught that these were module-private despite a comment promising `client.ts` reuse, and they were promoted to `export const` to match the already-exported type/status pair. In Phase 2 the same theme recurred: the `"Malformed devices page envelope"` string, hand-copied across the log line, `ProblemError.title`, and `detail`, became a single `MALFORMED_ENVELOPE_TITLE` constant, and the two Phase-2 log sites were routed through the shared `VALIDATION_ERROR_PREFIX` so all three `validation-error` sites are greppable on one head (which also made a previously-false `validation.ts` comment true).

**Making the "generic reuse" claim honest and the contracts unconditional.** Architect found `validateItems`' "Never throws" claim was only true in the `off` branch (a non-array reached `forEach` and threw a `TypeError` in `strict`/`warn`); the `Array.isArray` guard was hoisted above the mode switch so the contract holds unconditionally. Architect also flagged `extractIdentity`'s hardcoded `id`/`uid` extraction as a leaky abstraction relative to the advertised genericity, resolved with a doc comment stating the best-effort limitation.

**Test-hardening that pinned load-bearing contracts.** The implementation-auditor repeatedly tightened assertions to pin exactly the contracts the feature turns on: a bare `.toThrow()` became `.toThrow(ZodError)` (the type Phase 2's fatality check keys on); `warn`-path tests added a `console.warn` spy asserting it stays uncalled (the console→logger routing this feature fixes); `raw` was asserted `toBeInstanceOf(ZodError)` rather than merely defined; and Phase 2 gained direct coverage for all three R5 envelope-failure branches (non-object body, wrong-type `devices`, unparseable `pageDetails.nextPageUrl`) in validating modes, distinct from the `off` passthrough.

**Type-safety in the test suite.** typescript-cop replaced `as any` casts (one in Phase 1, seventeen in Phase 2) that bypassed `Result<T>`'s discriminated-union narrowing — the exact type the feature exists to make reliable. The Phase 2 fix introduced `assertOk`/`assertFail` assertion-signature helpers, which surfaced and closed real latent gaps the blanket cast had hidden (optional `warnings`, a stronger `"value" in r` check, a bound `find()` result).

## Key decisions

- **Fix the fixtures, not the schema.** The pre-existing fixture/schema drift could only be fixed by editing test data, since `DeviceSchema` is a protected file the plan forbids touching and the design gives no reason to loosen it. The reviser converted dates to epoch millis (preserving the represented instants) and filled `udf2`–`udf30` as `null`, with no assertion changes — the minimal, in-scope fix that made the phase's own gate pass.

- **The `identityOverride` parameter: added, then deliberately removed across phases.** Architect (Phase 1) added an optional `identityOverride?: string` to `toProblemError` so a single-value caller could inject a known identity instead of emitting a meaningless `index 0`; it was ratified as closing a genuine array-vs-single-value reuse gap. Project-lead immediately flagged it as speculative surface unused by the plan's own Phase 2 call site (ScopeCreep) — resolved in Phase 1 by keeping the parameter but documenting it as unused/optional rather than relitigating another reviewer's closed finding. In Phase 2, engineer confirmed it was dead across the whole codebase (`getDeviceByUid` never passed it, and the plan mandates `id=`-first identity as an accepted outcome), and its JSDoc falsely named `getDeviceByUid` as *the* consumer. The reviser then removed the parameter entirely, defaulted `index = 0` (dropping the meaningless bare `0` at the call site), deleted the false JSDoc, and left an inline comment recording the id-vs-uid tradeoff and pointing at the plan's rationale. The net effect: the speculative surface was retired once the phase that would have consumed it proved it wouldn't.

- **Keeping the phase notes honest.** Project-lead's findings were almost entirely about the durable `implementation-phase1-notes.md` going stale as revisions landed — first asserting the fixture fix was "out-of-scope"/unresolved after it had been fixed, then citing pre-revision test counts and claiming "No deviations" after four structural additions. Each was resolved by updating the notes to record the landed changes (as post-review addenda and documented deviations-from-snippet) so a Phase 2 implementor or historian reads the phase's final state, not its round-1 state.

- **Boundaries held throughout.** Every round re-verified that `schemas.ts`/`result.ts`/`index.ts` were byte-identical to `main`, that the internal envelope schema stayed off the barrel, and that no new top-level export leaked from `client.ts`/`config.ts` (R4 guards). `DevicesPageSchema` was kept exported and unchanged even though the runtime path stopped using it, per R4.

## Known limitations

- **Residual envelope gap, accepted by design.** A 200 body that is an object lacking *both* `pageDetails` and `devices` is treated as an empty page rather than a hard-fail. This is a documented, evidence-driven decision carried from the design (non-2xx responses never reach envelope validation, since `HttpClient` maps them to `{ ok: false }` first), pinned by dedicated tests — not a phase-introduced defect.

- **`internal/` module is deep-importable.** The un-barrelled `internal/devicesEnvelope.ts` ships in `dist/` and, absent a package `exports` map, is technically deep-importable. Architect noted this is a pre-existing property of the established un-barrelled-privacy pattern (shared with `validation.ts`), exposes no secret, and adding an `exports` map was out of this phase's scope (and would risk existing deep-import consumers).

- **Deferred consistency by design, not defect.** The `validate` single-value warn message body differs from the `validateItems`/`toProblemError` body because the single-value seam carries no `entityLabel`/identity — an inherent context difference that engineer considered and deliberately did not raise; only the greppable prefix is shared. Similarly, surfacing `strict`-mode rejections under the field named `warnings[]` is the design's settled `Result.warnings[]` channel, not a naming defect.
