## engineer — round 1

Plan Review Mode, round 1 (first engineer pass — the existing review-dir turns are from
plan-auditor/triage/reviser, not prior engineer turns, so no carry-forward applies). Reviewed the
plan against `design.md` and the real code it instruments: `src/http/http-client.ts`
(`handleResponseError`, `build403Error`, `buildRateLimitError`, retry branches), `src/auth/auth-manager.ts`
(`performRefresh`), `src/logging/logger.ts` (`dattoLoggerSchema`), and `src/http/axios-augment.d.ts`
(the `rateDescriptor` augment + the local `RetryTrackedConfig` intersection precedent). Axes walked in
order: DRY/reuse → abstraction/complexity → error-handling/logging → naming/intent → config/magic values.

Notable positives that are *not* findings: the LIFO interceptor-ordering reasoning is correct
(observer registered first → runs last → post-throttle, post-Bearer); the stash-reuse-over-`response.config`
rule is sound; the phase-scoped `declare module` exit gates are well placed.

| engineer-r1-f1 | High | Open | ErrorHandling | Phase 1 example `mapObserverError` (plan L168-176); shared-instance fire (Phase 2 S4, L256-260) | `mapObserverError` maps only `403 → build403Error`, else `DattoApiError.fromAxiosError`. But the client's actual terminal error for a **429 give-up** is `buildRateLimitError(waitMs, error)` (http-client.ts L299) — message "rate limit exceeded", populated `retryAfterMs`. Because `onError` fires unconditionally right after the `isAxiosError` guard (before all 429 handling), every 429 attempt is reported to the observer via `fromAxiosError`, which carries neither `retryAfterMs` nor the rate-limit message/semantics.
