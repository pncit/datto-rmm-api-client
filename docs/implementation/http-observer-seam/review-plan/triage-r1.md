### Dossier: plan-auditor-r1-f2

**History:** Raised in round 1 by `plan-auditor` as High/Consistency against a fresh (empty) review directory — no prior reviser disposition exists. The finding: the Phase 1 example doc comment contains "axios", while the Phase 1/Phase 4 exit gate greps the whole `http-observer.ts` file case-insensitively for "axios".
**Requirement at stake:** No Requirements-table R-ID is directly cited; the constraint is the design's Non-Goal "Exposing any axios type in the public contract" and the design exit-gate mandate (design line 260) that the observer's public surface stay axios-free. The gate is that mandate's enforcement.
**Reviewer's strongest case:** The example (plan line 96) literally instructs writing a comment containing "axios", and the gate (`! grep -iq 'axios' src/http/http-observer.ts`) is whole-file, case-insensitive — so any doc comment mentioning axios, including the word "axios-free" itself, makes the gate fail. As written, a verbatim-faithful implementation cannot pass its own exit gate: the phase is non-executable.
**Planner's strongest case:** The strict substring gate is a deliberately blunt guard against leaking an axios *import or type*; the "Opinionated Examples" section is illustrative, and a competent implementor would drop stray prose. The real intent — no axios import, no axios type in the signatures — is unambiguous, so the defect is cosmetic and cheaply fixed at implementation time.
**What you must decide:** Whether to (a) narrow the gate so it matches actual axios *import/type usage* rather than any substring, (b) strip the word "axios" from the example doc comment, or (c) both.
**Options:** 1) Narrow the gate — e.g. `! grep -Eq "from ['\"]axios['\"]" src/http/http-observer.ts` plus an optional `! grep -Eq '\bAxios[A-Z]' …` type check: precise, tolerates explanatory prose, still catches a real leak. 2) Only strip "axios" from the example comment: keeps the strict gate but remains fragile to any future axios-mentioning comment. 3) Both: precise gate *and* a clean example.
**Recommendation:** Option 3. The substring gate is a footgun independent of this one example — narrow it to an import/type match so honest prose can't trip it, and also remove the word from the example so the illustrative code and its gate agree. Resolution is a plan edit (both the gate blocks and the example live in `plan.md`).

### Dossier: plan-auditor-r1-f3

**History:** Raised in round 1 by `plan-auditor` as High/DesignAlignment against a fresh review directory — no prior reviser disposition. Plan Phase 1 S3 routes all five public observer types through `src/index.ts` directly and explicitly does not touch `public-types.ts`, citing the `DattoLogger` precedent (verified in the repo: `index.ts` exports `DattoLogger` directly and `public-types.ts` carries generated/reconciled entity re-exports). **Reviewer's strongest case:** The plan diverges from the design's literal, stated export location; a plan should either conform to the design or force an explicit design amendment, not silently deviate. Leaving the deviation undispositioned invites drift between design and implementation.
**Planner's strongest case:** Routing hand-authored public types through `index.ts` matches the verified `DattoLogger` precedent, while `public-types.ts` is reserved for generated/reconciled entity types — so the `index.ts`-direct placement is the cleaner, convention-consistent option and the design's line 267 is the imprecise party.
**What you must decide:** Either add the five observer types to `public-types.ts` per the design's literal statement, or accept the `index.ts`-direct placement and amend design line 267 (and 87/95) to match.
**Options:** 1) Add the five types to `public-types.ts` — literal design compliance, but breaks the established `DattoLogger` convention for hand-authored types. 2) Accept `index.ts`-direct and amend the design to say the observer types export directly from `index.ts` alongside `DattoLogger` — matches the verified precedent; requires a design amendment.
**Recommendation:** Option 2. The `index.ts`-direct placement is consistent with the repo's own verified `DattoLogger` precedent and yields an identical package-root surface; the cleaner fix is to correct design line 267 rather than bend the plan to a less consistent convention. Because this is High, the call is yours; if you agree, the resolution flows as a `design` amendment.
