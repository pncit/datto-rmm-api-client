## Findings

| ID | Severity | Status | Category | Location | Finding | Recommendation |
|----|----------|--------|----------|----------|---------|----------------|
| project-lead-r1-f1 | High | Open | BehaviorIntent | `src/http/http-observer.ts:120-131` (`observerCallbackSchema` / `dattoHttpObserverSchema`); `src/client/datto-client-config.ts:103` (`DattoRmmClientConfig = z.infer<...>`) | The round-1 fix for the wrapping-`z.function` defect replaced it with **one shared** `z.custom<(event: never) => unknown>` reused for all three callback fields. This repairs R7/R9 at the helper level, but it collaterally regresses the **public config type**: `DattoRmmClientConfig` is `z.infer<typeof dattoRmmClientConfigSchema>`, so the exported type of `httpObserver.onRequest`/`onResponse`/`onError` is literally `(event: never) => unknown`, not the friendly `DattoHttpObserver` interface with its three named, concretely-typed events. Verified two ways: (1) `dist/index.d.ts:33-35` shows `onRequest: z.ZodOptional<z.ZodCustom<(event: never) => unknown, ...>>` etc.; (2) a scratch consumer file calling `createDattoRmmClient({ ..., httpObserver: { onRequest: (event) => { console.log(event.method); } } })` — the single most natural/idiomatic way to supply the observer, and the exact shape shown in this plan's own Phase-1/Phase-4 snippets minus a pre-typed intermediate variable — fails `tsc --noEmit --strict` with `Property 'method' does not exist on type 'never'` because `event` is contextually inferred as `never`. A consumer only avoids this by manually pre-declaring `const observer: DattoHttpObserver = {...}` before passing it in (verified this workaround compiles clean) — an extra, undocumented step nothing in Phase 1 signals is required. This undermines R1's Goal and the design's stated purpose of exporting three named, concretely-typed event interfaces: for the default/idiomatic usage pattern, none of that typing reaches the callback parameter. Note the existing `logger` field does not have this problem — `dattoLoggerSchema`'s per-method `z.function({ input: [...] })` preserves concrete parameter types through `z.infer`, so `httpObserver` is a regression relative to that established precedent, not merely equivalent to it. | Give each callback field its own precisely-typed validator instead of one field-agnostic `(event: never) => unknown` custom schema, e.g. a small generic `const observerCallbackSchema = <E,>() => z.custom<(event: E) => unknown>((v) => typeof v === "function").optional();` used as `onRequest: observerCallbackSchema<DattoHttpRequestEvent>()`, `onResponse: observerCallbackSchema<DattoHttpResponseEvent>()`, `onError: observerCallbackSchema<DattoHttpErrorEvent>()` — this keeps the non-wrapping `z.custom` fix (R7/R9 stay satisfied) while restoring `z.infer`'s per-field precision so `DattoRmmClientConfig.httpObserver` structurally matches `DattoHttpObserver`. (Alternatively, override just this one field on the exported type: `type DattoRmmClientConfig = Omit<z.infer<typeof dattoRmmClientConfigSchema>, "httpObserver"> & { httpObserver?: DattoHttpObserver };` — more of a patch, and it would make the type's own doc comment claim of being wholly schema-inferred no longer strictly true.) Add a compile-only regression test (e.g. in `surface-pin.ts` or a dedicated type-test) asserting an inline `httpObserver: { onRequest: (event) => event.method }` object literal passed straight to `createDattoRmmClient`/`DattoRmmClientConfig` type-checks without an explicit annotation, so this cannot silently regress again. |
