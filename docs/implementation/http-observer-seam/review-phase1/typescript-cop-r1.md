## typescript-cop — round 1

Scope: Phase 1 of the HTTP observer seam — `src/http/http-observer.ts` (five public types +
`dattoHttpObserverSchema`), `src/http/observer.ts` (internal helper), `src/http/axios-augment.d.ts`
(stash augment), `src/client/datto-client-config.ts` (strict-config field), `src/index.ts` (public
export), and the accompanying tests. Reviewed the current working tree (which already includes the
round-1 fix for `implementation-auditor-r1-f1`/f2/f3 — the `z.function` → `z.custom` schema swap —
ratified by `implementation-auditor-r2`). This is `typescript-cop`'s first round on this phase; no
prior `typescript-cop` turn exists to reconcile.

The five public types themselves (`DattoHttpHeaders`/`DattoHttpRequestEvent`/`DattoHttpResponseEvent`/
`DattoHttpErrorEvent`/`DattoHttpObserver`) are clean, axios-free, and correctly typed. The defect is in
how `dattoHttpObserverSchema` is built: it reuses one shared, single-generic-typed callback schema for
all three differently-shaped callbacks, which — confirmed by building `dist/index.d.ts` and by an
independent `tsc` check — corrupts the *directly-exported* `DattoRmmClientConfig` type's `httpObserver`
field. `src/http/observer.ts`'s `invokeObserver` uses a related `never`-erasure trick internally that,
while currently harmless (all three call sites are hand-written correctly), defeats compile-time
checking of the callback/event pairing for no remaining benefit once f1 is fixed, and its own
swallow-`warn` calls are unguarded against a throwing `logger`.

| typescript-cop-r1-f1 | High | Open | Public Types & Export Hygiene | `src/http/http-observer.ts:120-134` (`observerCallbackSchema`/`dattoHttpObserverSchema`); observed via `dist/index.d.ts:32-37` and `src/client/datto-client-config.ts:81-85` (`DattoRmmClientConfig`) | `observerCallbackSchema` is `z.custom<(event: never) => unknown>(...)`, reused unchanged for `onRequest`, `onResponse`, and `onError` alike. Because all three fields share the *same* generic type argument, `z.infer<typeof dattoHttpObserverSchema>` — and therefore `DattoRmmClientConfig['httpObserver']` (`z.infer<typeof dattoRmmClientConfigSchema>`, exported directly from `src/index.ts`) — types every callback as `(event: never) => unknown` instead of `DattoHttpObserver`'s concrete per-field event types. Confirmed in the built output: `dist/index.d.ts:33-35` shows `onRequest: z.ZodOptional<z.ZodCustom<(event: never) => unknown, (event: never) => unknown>>` (and identically for `onResponse`/`onError`). Independently verified with `tsc --strict`: a value of `DattoRmmClientConfig['httpObserver']` cannot have its `onRequest` invoked with a real `DattoHttpRequestEvent` — `tsc` reports "Argument of type 'DattoHttpRequestEvent' is not assignable to parameter of type 'never'." This is not hypothetical: `tests/unit/client/config.test.ts:65` already has to route around it with an unsafe `rawEvent as never` cast just to invoke the parsed callback under test. A consumer who types a variable as `DattoRmmClientConfig` (the published config type) and later tries to read/forward/invoke `config.httpObserver.onRequest` hits the same wall — an internal type-erasure trick used to make one non-generic runtime validator reusable has leaked into a directly-published type, making that field's declared shape unusable for its intended purpose. | Give each callback field its own concrete type argument instead of sharing one. E.g.: `function callbackSchema<Fn>() { return z.custom<Fn>((v) => typeof v === "function").optional(); }` then `z.strictObject({ onRequest: callbackSchema<DattoHttpObserver["onRequest"]>(), onResponse: callbackSchema<DattoHttpObserver["onResponse"]>(), onError: callbackSchema<DattoHttpObserver["onError"]>() })`. Verified this compiles cleanly and makes `z.infer<typeof dattoHttpObserverSchema>.onRequest` callable with a real `DattoHttpRequestEvent` with no cast, while preserving the non-wrapping runtime behavior (identity pass-through, no return-value validation) that the round-1 `z.function`→`z.custom` fix already established. Remove the now-unnecessary `as never` cast in `config.test.ts:65` once fixed. |
| typescript-cop-r1-f2 | Medium | Open | Generics & Inference | `src/http/observer.ts:92-120` (`invokeObserver`) | `invokeObserver`'s `fn` parameter is typed `((event: never) => void) \| undefined`, and the body immediately casts it away — `(fn as (event: unknown) => unknown)(event)` — to actually call it. This double erasure (concrete callback type → `never` at the parameter, then `never` → `unknown` via cast at the call) means nothing checks that the callback passed in (`observer.onRequest`/`onResponse`/`onError`) matches the `event` object built for it at each of the three call sites (`fireRequest`/`fireResponse`/`fireError`); swapping which callback or which event is passed at any one of those sites would still compile. The `never` parameter is unnecessary once a plain generic is used, and the internal `as` cast can be eliminated entirely. | Make `invokeObserver` generic over the event type: `function invokeObserver<E>(logger: DattoLogger \| undefined, callbackName: ObserverCallbackName, fn: ((event: E) => void) \| undefined, event: E): void`. Verified this compiles without any internal cast — `const returned: unknown = fn(event);` typechecks directly (a callback statically returning `void` is still assignable to a `const` typed `unknown`) — and each call site becomes `invokeObserver<DattoHttpRequestEvent>(logger, "onRequest", observer.onRequest, event)` (similarly for `onResponse`/`onError`), so the compiler now checks that the right callback is paired with the right event shape at each of the three sites. |
| typescript-cop-r1-f3 | Medium | Open | Async/Await Correctness | `src/http/observer.ts:101-119` (`invokeObserver`'s `catch` block and the `.then(undefined, ...)` rejection handler) | Both swallow paths call `logger?.warn(...)` unguarded. Nothing in `DattoLogger`'s type or `dattoLoggerSchema`'s shape-only validation prevents a consumer-supplied `warn` implementation from itself throwing. If it does: (a) in the synchronous `catch` block, that new exception propagates straight out of `invokeObserver` — into the interceptor/grant call path — which is exactly the outcome `invokeObserver` exists to prevent (R7: "never alter, delay, or fail the request"); (b) in the `.then(undefined, () => logger?.warn(...))` handler, a throwing `warn` rejects the promise `.then` returns, and that promise is never stored, awaited, or chained further — a floating promise whose rejection becomes an **unhandled rejection**, which by default Node.js behavior terminates the process. Both cases are the *observer plumbing itself* reintroducing the exact failure mode (an uncontrolled throw/unhandled rejection from third-party-ish code) that this module's entire purpose is to eliminate. | Wrap each `logger?.warn(...)` call in its own nested `try { … } catch { /* logger itself misbehaved; nothing further to do safely */ }` (both in the synchronous `catch` block and inside the `.then` rejection callback), so a throwing logger can never escape `invokeObserver` or leave an unhandled rejection behind. |
